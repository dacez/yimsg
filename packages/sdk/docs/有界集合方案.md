## 1. 目标与原则

Yimsg SDK 长期运行时集合必须有硬上限，因此所有长期驻留集合都建立在「容量在构造时指定、size 永不超过 capacity」的有界结构之上：

- 所有长期驻留集合必须有硬上限，禁止无界 Map / Set / Queue 长期增长。
- `BoundedU64Set` / `BoundedQueue` 基于定长 TypedArray，额外提供「运行期字节上界可静态精确计算」的强保证，结构上开放寻址 + 固定桶 + 桶内线性扫描，易于跨语言（Rust / Go / C）复刻。
- `FifoU64Map` 放弃了「字节上界静态精确计算」这一强保证，改用原生 `Map` 实现，仅保留「entry 数量有界 + FIFO 淘汰」，换取实现简单与任意值类型支持（详见第 2 节）。
- 网络协议整包最大仍为 64KB（见协议帧约束），与有界集合容量共同构成内存上界。

### 1.1 Message 不长期驻留

SDK 内存中**不允许**保存 `msg_id -> Message` 的长期全局缓存 Map。Message 属于「窗口态数据」：当前可见窗口、preload window、正在发送、短期 dedup。长期消息存储依赖持久层（SQLite / IndexedDB / OPFS）。增量同步单批最多 `DEFAULT_SYNC_BATCH_SIZE`（200）条，派发后立即释放，不累积。

## 2. FifoU64Map\<V\>

容量受限的 `uint64 -> V` FIFO 映射，泛型值类型默认 `V = string`。

### 2.1 uint64 key

key 统一为十进制 uint64 字符串（与项目内 uid / group_id / request_id 的惯例一致）。内部通过 `u64.ts` 的 `parseU64` / `u64ToString` 做严格校验与规范化：非法字符串（负数、小数、空串、非数字、越界）直接抛 `RangeError`；"007" 与 "7" 规范化为同一 key，避免字符串字面量差异导致的重复条目。

### 2.2 实现与淘汰策略

内部直接基于原生 `Map<string, V>` 实现：

- `capacity` 在构造时指定，`size` 永不超过 `capacity`。
- 写入新 key 且 `size` 超出 `capacity` 时，淘汰最早插入的 key（FIFO，不支持 reject / lru）。
- 更新已存在 key 的 value 不改变其插入顺序位置（原生 `Map.set()` 语义），因此重复写入同一 key 不会重置其 FIFO 顺序。
- 不做 bucket 对齐、不预分配 TypedArray，因此不再具备「运行期字节上界可静态精确计算」的强保证；`estimateFifoU64MapBytes` 给出的是「容量 × (Map 条目结构开销 + 值字节)」的粗略上界估算。

## 3. BoundedU64Set

固定容量 uint64 集合，基于开放寻址 + 固定桶 + 桶内线性扫描布局（`bucket-layout.ts` 提供容量对齐算法），不保存 value，专用于「待拉取 / 在飞」这类去重队列。默认 reject 策略：容量满时 `add()` 返回 `false`，不允许新 key 淘汰已有 key。提供 `drain()` 原子排空。

## 4. BoundedQueue\<V\>

固定容量环形缓冲 FIFO 队列：`enqueue` / `dequeue` / `peek` / `clear` / `size` / `capacity`。两种溢出策略：

- `reject`：队满时 `enqueue()` 返回 `false`，保留旧元素（默认）。
- `overwrite_oldest`：队满时丢弃队首最旧元素再写入。

## 5. 运行时统计

每个有界集合都暴露 `stats(): BoundedStats`：`size`、`capacity`、`bucketCount`、`bucketCapacity`、`rejectCount`、`evictionCount`、`loadFactor`。非 bucket 结构（`FifoU64Map`、`BoundedQueue`）统一以 `bucketCount=1`、`bucketCapacity=capacity` 语义暴露；`FifoU64Map` 从不拒绝写入（capacity=0 除外），故 `rejectCount` 恒为 0。`YimsgClient.getBoundedCollectionStats()` 聚合显示信息缓存与待响应请求的统计，用于 benchmark / debug / 回归监控。

## 6. 第一阶段承载的集合

| 集合 | 承载结构 | 策略 | 说明 |
|------|---------|------|------|
| `pendingRequests`（WsTransport） | `FifoU64Map<PendingReq>` | fifo | key = request_id；`sendBinary()` 在 `size ≥ maxPendingRequests` 时先拒绝，因此底层 FIFO 淘汰在正常路径下不会触发，仅作兜底 |
| 用户显示信息缓存 | `FifoU64Map<DisplayCacheEntry>` | fifo | key = uid |
| 群显示信息缓存 | `FifoU64Map<DisplayCacheEntry>` | fifo | key = group_id；与用户缓存完全独立 |
| 显示信息待拉取 / 在飞队列 | `BoundedU64Set`（用户/群各 2 个） | reject | 满则抛 `ValidationError`，用户与群上限独立 |

用户与群缓存严格拆分（不使用 `kind + id` 统一 map），允许少量容量浪费，换取极简结构、纯 uint64 key、无 uid/group_id 冲突。

**未纳入**（明确禁止）：全局 Message Map、无限 Message Cache。

## 7. 峰值内存估算

`YimsgClient.estimateMaxMemoryBytes(options)` 把所有有界集合纳入理论峰值估算。每个有界集合满载字节由 `estimateFifoU64MapBytes` / `estimateBoundedU64SetBytes` / `estimateBoundedQueueBytes` 计算：

```text
totalBytes
  = profileUserCacheBytes   // 用户显示信息 FifoU64Map(fifo) 满载
  + profileGroupCacheBytes  // 群显示信息 FifoU64Map(fifo) 满载
  + profileQueueBytes       // 4 × 待拉取/在飞 BoundedU64Set 满载
  + pendingRequestsBytes    // 待响应请求 FifoU64Map 满载
  + syncBatchBytes          // DEFAULT_SYNC_BATCH_SIZE × BYTES_PER_SYNC_MESSAGE（常量）
  + forwardBundleBytes      // DEFAULT_MAX_FORWARD_BUNDLE_BYTES（常量）
  + baselineBytes           // SDK_BASELINE_BYTES（常量）
```

容量参数集中在 `sdk-defaults.ts`（`DISPLAY_CACHE_BUCKET_CAPACITY`、`DISPLAY_QUEUE_LOAD_FACTOR`、`BYTES_PER_DISPLAY_CACHE_VALUE`、`BYTES_PER_PENDING_REQUEST_VALUE` 等）。`FifoU64Map` 不再需要 bucket 布局参数。
