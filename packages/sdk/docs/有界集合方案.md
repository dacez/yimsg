## 1. 目标与原则

Yimsg SDK 长期运行时集合必须有硬上限，因此所有长期驻留集合都建立在「容量在构造时指定、size 永不超过 capacity」的有界结构之上：

- 所有长期驻留集合必须有硬上限，禁止无界 Map / Set / Queue 长期增长。
- `FifoMap<K,V>` / `FifoSet<T>` 基于原生 `Map` / `Set` 实现，key / value（或元素）均为任意类型，放弃了「运行期字节上界可静态精确计算」这一强保证，只保留「entry 数量有界 + FIFO 淘汰」，换取实现简单与完全通用性（详见第 2、3 节）。
- `BoundedQueue` 基于定长数组环形缓冲，额外提供「运行期字节上界可静态精确计算」的强保证。
- 网络协议整包最大仍为 64KB（见协议帧约束），与有界集合容量共同构成内存上界。

### 1.1 Message 不长期驻留

SDK 内存中**不允许**保存 `msg_id -> Message` 的长期全局缓存 Map。Message 属于「窗口态数据」：当前可见窗口、preload window、正在发送、短期 dedup。长期消息存储依赖持久层（SQLite / IndexedDB / OPFS）。增量同步单批最多 `DEFAULT_SYNC_BATCH_SIZE`（200）条，派发后立即释放，不累积。

## 2. FifoMap\<K, V\>

容量受限的通用 FIFO 映射，key / value 均为任意类型（无默认值，无 uint64 限定）。

### 2.1 key / value 语义

key 与 value 的相等性、哈希与原生 `Map` 完全一致（SameValueZero）：基本类型按值比较，对象按引用比较。不再局限于十进制 uint64 字符串，也不做任何规范化或校验——项目内实际使用的 key 类型包括十进制字符串（uid / group_id / request_id / msg_id / convKey）与 number（`SqliteWorkerApi` 的 worker RPC id）。

### 2.2 实现与淘汰策略

内部直接基于原生 `Map<K, V>` 实现：

- `capacity` 在构造时指定，`size` 永不超过 `capacity`。
- 写入新 key 且 `size` 超出 `capacity` 时，淘汰最早插入的 key（FIFO，不支持 reject / lru）。
- 更新已存在 key 的 value 不改变其插入顺序位置（原生 `Map.set()` 语义），因此重复写入同一 key 不会重置其 FIFO 顺序。
- 实现 `[Symbol.iterator]`（产出 `[key, value]`），与原生 `Map` 一样支持 `for...of` 与展开运算符 `[...map]`。
- 不做任何预分配，因此不再具备「运行期字节上界可静态精确计算」的强保证，也不提供内存估算函数。

## 3. FifoSet\<T\>

容量受限的通用 FIFO 去重集合，元素为任意类型，与 `FifoMap` 同源设计（基于原生 `Set` 实现，同样的容量/淘汰/迭代语义），额外提供 `drain()`：取出全部元素（FIFO 顺序）并原子清空，用于「待拉取」队列排空。

## 4. BoundedQueue\<V\>

固定容量环形缓冲 FIFO 队列：`enqueue` / `dequeue` / `peek` / `clear` / `size` / `capacity`。两种溢出策略：

- `reject`：队满时 `enqueue()` 返回 `false`，保留旧元素（默认）。
- `overwrite_oldest`：队满时丢弃队首最旧元素再写入。

## 5. 运行时统计

每个有界集合都暴露 `stats(): BoundedStats`：`size`、`capacity`、`bucketCount`、`bucketCapacity`、`rejectCount`、`evictionCount`、`loadFactor`。非 bucket 结构（`FifoMap`、`FifoSet`、`BoundedQueue`）统一以 `bucketCount=1`、`bucketCapacity=capacity` 语义暴露；`FifoMap` / `FifoSet` 从不拒绝写入（capacity=0 除外），故 `rejectCount` 恒为 0。`YimsgClient.getBoundedCollectionStats()` 聚合显示信息缓存与待响应请求的统计，用于 benchmark / debug / 回归监控。

## 6. 承载的集合

| 集合 | 承载结构 | 说明 |
|------|---------|------|
| `pendingRequests`（WsTransport） | `FifoMap<string, PendingReq>` | key = request_id；`sendBinary()` 在 `size ≥ maxPendingRequests` 时先拒绝，因此底层 FIFO 淘汰在正常路径下不会触发，仅作兜底 |
| 用户显示信息缓存 | `FifoMap<string, DisplayCacheEntry>` | key = uid |
| 群显示信息缓存 | `FifoMap<string, DisplayCacheEntry>` | key = group_id；与用户缓存完全独立 |
| tag_id → org_id 旁路记录（DisplayInfoCache） | `FifoMap<string, string>` | 容量与待拉取队列同阶，非强一致 |
| 显示信息待拉取 / 在飞队列 | `FifoSet<string>`（用户/群/org/tag 各 2 个） | 上层 `enqueue()` 显式前置容量检查，满则抛 `ValidationError`；各域上限独立 |
| `BaseDataGateway` 的 5 个通知累积器（`pendingClearedKeys`/`pendingDeletedConvKeys`/`pendingDeletedMsgIds`/`pendingMessageIds`/`pendingOrgIds`） | `FifoSet<string>` / `FifoMap<string,string>`，容量均为 `batchMaxLimit` | 高频通知在 flush 前的去重累积；容量满时 FIFO 淘汰最旧 key，只是延迟到下一批处理，不影响最终一致性（重绘信号兜底） |
| `SqliteWorkerApi.pending`（persistent 模式 OPFS worker RPC 关联表） | `FifoMap<number, {resolve,reject}>`，容量 `DEFAULT_SQLITE_WORKER_MAX_PENDING_CALLS` | key = 自增调用 id；`call()` 在 `size ≥ 上限` 时立即 reject，同样保证底层 FIFO 淘汰不会在正常路径触发 |

用户与群缓存严格拆分（不使用 `kind + id` 统一 map），允许少量容量浪费，换取极简结构、无 uid/group_id 冲突。

**未纳入**（明确禁止）：全局 Message Map、无限 Message Cache；以及事件监听表、UI 选择状态、临时表单键盘等「必须精确、不可静默丢弃」或「天然由小型固定枚举限定规模」的 Map/Set——这些不适合套用 FIFO 淘汰语义，继续使用原生 Map/Set。
