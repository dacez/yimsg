Yimsg 的 WebSocket 推送只承担“状态变化信号”职责。服务端在进程内以强类型 `appmsg.Notification` 传递通知，出站时直接编码为 protobuf notification frame；客户端收到后通过 SDK 主动同步或更新本地缓存。跨消息、通讯录、屏蔽列表、免打扰和会话的同步细节统一见 [`docs/architecture/同步机制方案.md`](../../docs/architecture/同步机制方案.md)。

---

## 1. 核心原则

| 原则 | 说明 |
|------|------|
| 信号优先 | 推送只告诉客户端“有变化”，同步副本通过 `sync_*` 追平，界面按读模型分页重读 |
| 多端同步 | 同一用户多个在线连接都会收到与自身相关的通知 |
| 幂等消费 | 客户端收到重复通知时，应以服务端同步结果为准 |
| 全链路 protobuf | 进程内通知是结构体，WebSocket 出站是 protobuf frame，不经过 JSON 序列化 / 反序列化中转 |

---

## 2. Notification 结构

`server/internal/appmsg/notification.go` 定义核心通知结构：

```go
type Notification struct {
  Type   string
  Target *ConversationTarget
  MsgID  string
}
```

通知在服务端内部通过 `online.Conn.Ch chan *appmsg.Notification` 传递；`server/internal/ws/connection.go` 读取结构体后调用 `notificationToProto()` 与生成的 `EncodeNotificationFrame()` 直接写出 protobuf frame。核心通知链路不再定义 `Data json.RawMessage`，也不再支持通过通知携带任意 JSON 扩展字段；复杂扩展数据应通过 protobuf action 拉取。

## 3. 核心通知类型

| type | 触发场景 | 关键字段 | 客户端处理 |
|------|----------|----------|------------|
| `messages:received` | 新消息、撤回最终态等消息流变化 | `target`（`from_uid`/`group_id`）, `msg_id`（触发本次通知的消息 id，合并通知时取最新一条） | 纯重绘信号：SDK 把同一调度窗口内各通知的 `msg_id` 累积去重后用 `get_messages(msg_ids=[...])` 批量拉取内容供 `onMessages`（角标/响铃）；持久存储先 `sync_messages` 写入本地再回调；UI 收到信号后用 `get_conversations`/`get_messages` 重绘，不消费通知 payload |
| `contacts:updated` | 好友申请、接受、拒绝、删除、联系人 `sort_key`/`search_text` 投影刷新 | 无 | instant 发失效事件后由 UI 重读 `get_contacts`；持久存储调用 `sync_contacts` 追平联系人域 |
| `blocklist:updated` | 屏蔽列表新增或解除 | 无 | instant 发失效事件后由 UI 重读过滤分页状态；持久存储调用 `sync_blocklist` 追平屏蔽列表域 |
| `conversations:mutelist-updated` | 会话免打扰状态变化 | 无 | instant 发失效事件后由 UI 重读过滤分页状态；持久存储调用 `sync_mutelist` 追平免打扰域 |
| `session:kicked` | 当前 token 被其他设备或登出逻辑踢下线 | 无 | 清理会话并提示重新登录 |
| `conversations:clearunread` | 同用户其他设备清除会话未读 | `target`（`from_uid?`/`group_id?`） | 只在数据窗口内就地把该会话未读清零，**绝不触发拉取**；不在窗口则忽略（不可见，靠后续全量刷新追平） |
| `conversations:delete` | 同用户其他设备删除会话 | `target`（`from_uid?`/`group_id?`） | 命中数据窗口则就地删除该会话、剩余条目往上补齐（防抖动），**绝不触发拉取**；不在窗口则忽略 |
| `messages:delete` | 同用户其他设备删除消息 | `target`（`from_uid?`/`group_id?`）, `msg_id` | 命中数据窗口则就地删除该消息、剩余消息往上补齐，**绝不触发拉取**；不在窗口则忽略 |
| `org:updated` | 组织架构变更（建 tag、连边、调 rank、改名、入离职等），经 taskqueue 向全体在线成员扇出、队列内按组织合并；管理员授权/撤权（GRANT 边）不触发这个通知——GRANT 边本就不出现在 `get_tags`/`sync_tags` 结果里，扇出对普通成员是无意义的空变更。`delete_org` 是例外：不走 taskqueue，用删除前的成员快照直接同步推送（异步任务执行时组织结构已经查不到成员） | `org_id`（仅提示，禁止携带增量数据） | instant 发失效事件后由 UI 重拉展开数据；持久存储按 `sync_tags(org_id)` 增量追平该组织本地副本；`get_org_infos`/`get_tags` 均查不到时说明组织已被删除，清空本地该组织的副本 |

## 4. 触发矩阵

| 服务端动作 | 推送类型 | 接收者 | 说明 |
|------------|----------|--------|------|
| `send_message` 私聊 | `messages:received`（携带新消息 `msg_id`） | 发送者与接收者全部在线连接 | 发送者本端以 action 响应更新；其他在线端按累积的 `msg_id` 批量拉内容供 `onMessages` 并重绘，持久存储先同步本地 |
| `send_message` 群聊 | `messages:received`（携带新消息 `msg_id`） | 所有群成员与发送者在线连接 | 发送者 inbox 写入后立即返回；其余成员异步扇出完成后通知客户端，扇出通知携带该批 `msg_id` |
| `send_message`(`MESSAGE_TYPE_RECALL`) | `messages:received`（携带 recall 事件 `msg_id`） | 受影响会话成员 | UI 按信号重读 `get_messages`，撤回占位与事件由服务端最新态统一反映；`onMessages` 收到 recall 事件消息 |
| `add_friend` | `contacts:updated` | 被申请方 | 客户端刷新待处理请求列表 |
| `accept_friend` / `reject_friend` | `contacts:updated` | 关系双方 | 客户端刷新联系人或请求列表 |
| `delete_friend` | `contacts:updated` | 调用方在线连接 | 单向删除好友或取消群收藏，仅调用方多端刷新通讯录 |
| `get_user_infos` / `get_group_infos` | `contacts:updated` | 调用方在线连接 | 当联系人/收藏群的 `sort_key`/`search_text` 因昵称或群名变化而刷新时发送 |
| `block_user` / `unblock_user` | `blocklist:updated` | 操作者在线连接 | 客户端刷新屏蔽列表状态；屏蔽时若清理本侧待处理好友请求，会额外发送 `contacts:updated` |
| `mute_conversation` / `unmute_conversation` | `conversations:mutelist-updated` | 操作者在线连接 | 客户端刷新会话免打扰偏好 |
| `logout` / `kick` | `session:kicked` | 被失效 token 对应连接 | 客户端进入未登录态 |
| `delete_message` | `messages:delete`（携带 `msg_id`） | 操作者其他在线连接 | 命中数据窗口就地删消息（不拉取）；持久存储删本地行 |
| `delete_conversation` | `conversations:delete` | 操作者其他在线连接 | 命中数据窗口就地删会话（不拉取）；持久存储删本地行 |
| `clear_unread` | `conversations:clearunread` | 同 uid 的其他在线连接 | 多端就地清未读红点（不拉取）；服务端只清未读不动 seq，会话行已 GC 时不重建，等下一条消息到来自然追平 |
| `create_org_tag` / `rename_org_tag` / `delete_org_tag` / `link_org_tag` / `add_org_member` / `remove_org_member` / `set_org_item_rank` / `rename_org` | `org:updated`（携带 `org_id`） | 组织全体在线成员 | 只提示某组织架构变化，不带增量数据；客户端按 `org_id` 重拉 `get_tags`/`get_tag_infos`/`sync_tags` 追平（`grant_org_admin`/`revoke_org_admin` 只变更管理员授权、不出现在 `get_tags` 结果里，不触发 `org:updated`） |
| `delete_org` | `contacts:updated` | 各成员自己 uid 分片的在线连接 | 组织结构在写事务内同步清空；每个成员通讯录组织行经 taskqueue 异步 tombstone 后逐个推送，避免大组织阻塞删除请求 |

## 5. 客户端事件桥接

`packages/sdk/src/client-event-bridge.ts` 负责把原始通知转换为 SDK 事件，并交给 DataGateway 处理；DataGateway 的队列合并、instant 直读和持久存储同步差异见 [`docs/architecture/同步机制方案.md`](../../docs/architecture/同步机制方案.md)：

同步域通知和 SDK 公开事件尽量保持同名，以降低概念切换；同名只表示所属同步域一致，具体 payload 以所在层协议为准。

| 通知类型 | SDK 事件 | 行为 |
|----------|----------|------|
| `messages:received` | `messages:received`（重绘信号；payload `messages` 承载按累积的 `msg_id` 批量取到的内容供 `onMessages`） | instant 按累积的 `msg_id` 批量直读内容，持久存储先同步本地副本再读；UI 收到信号用 `get_conversations`/`get_messages` 重绘，不消费 payload |
| `contacts:updated` | `contacts:updated` | instant 直接发失效事件，持久存储同步联系人变化后触发事件；通讯录可见时重新拉取当前页 |
| `blocklist:updated` | `blocklist:updated` | 通知 UI 重读屏蔽列表快照或持久存储增量 |
| `conversations:mutelist-updated` | `conversations:mutelist-updated` | 通知 UI 重读会话免打扰偏好或持久存储增量 |
| `session:kicked` | `session:kicked` | 清理认证态 |
| `conversations:clearunread` | `conversations:clearunread`（`{ keys }`） | 持久存储先清本地未读再回调（sync-first）；UI 对仍在数据窗口的会话 `getConversations({ targets })` 定向拉取当前状态并更新窗口，**不整列表重拉**；不在窗口则忽略 |
| `conversations:delete` | `conversations:delete`（`{ keys }`） | 持久存储先删本地行再回调；UI 定向拉取（返回空=已删）→ 从窗口移除并往上补齐 |
| `messages:delete` | `messages:deleted`（`{ messageId, key }`） | 持久存储先删本地消息行再回调；UI 消息窗口就地删除，并对会话 `key` 定向刷新预览 |

本端发起的 `delete_conversation` / `delete_message` 也走同一条 `conversations:delete` / `messages:delete` 路径（SDK 内部本地触发），保证本端与他端行为一致。

UI 层只订阅 SDK 事件，不直接消费原始 WebSocket 包。

---

## 6. 测试建议

- 后端协议测试覆盖 typed notification 与 protobuf frame 的互转，确保 `msg_id`、`target` 等字段不丢失。
- E2E 测试覆盖消息、联系人、会话踢下线、清未读同步等核心通知。
- 新增通知类型时，应同步 `protocol/yimsg.proto`、生成物、SDK 事件桥接和本文触发矩阵。
