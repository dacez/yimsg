本文作为关系拦截、会话提醒偏好和红点语义的单一事实源。

## 1. 范围与目标

| 能力 | 当前目标 | 非目标 |
|---|---|---|
| 屏蔽列表 | 用户单向设置屏蔽列表；任一方向屏蔽后冻结私聊和好友申请主动链路；多端同步 | 群级屏蔽列表、资料可见性策略、管理后台审计 |
| 会话免打扰 | 按单聊 / 群聊会话设置静音；消息照常投递，会话照常刷新，但静音会话不递增未读数 | 全局勿扰、定时静音、仅 @ 提醒、系统 Push 完整策略 |
| 红点 / 未读 | 服务端维护会话未读；前端以会话分页和待处理请求分页驱动导航红点 | 前端全量未读 KV、启动全量屏蔽列表 / 免打扰快照 |

核心原则：屏蔽列表是访问控制，免打扰是提醒策略，红点是服务端会话摘要和联系人待处理请求的展示结果，三者不互相复用状态字段。

## 2. 数据模型

### 2.1 屏蔽列表

屏蔽列表独立于联系人表：

```sql
CREATE TABLE IF NOT EXISTS blocklist (
    uid         INTEGER NOT NULL,
  block_uid   INTEGER NOT NULL,
    status      INTEGER NOT NULL DEFAULT 0,
    seq         INTEGER NOT NULL DEFAULT 0,
    created_at  INTEGER NOT NULL,
    updated_at  INTEGER NOT NULL,
    PRIMARY KEY (uid, block_uid)
);
  CREATE INDEX IF NOT EXISTS idx_blocklist_seq ON blocklist(uid, seq);

  CREATE TABLE IF NOT EXISTS blocklist_version (
  uid         INTEGER PRIMARY KEY,
  gc_safe_seq INTEGER NOT NULL DEFAULT 0,
  max_seq     INTEGER NOT NULL DEFAULT 0
);
```

`status=1` 表示生效，`status=0xff` 表示解除后的 tombstone。解除屏蔽不立即物理删除，保证其他设备可通过 `sync_blocklist` 感知删除；后台 Blocklist GC 会按批清理已解除记录并推进 `blocklist_version.gc_safe_seq`，持久存储过旧游标收到 `seq_too_old` 后清空本地屏蔽列表表和游标，再从 `last_seq=0, rebuild=true` 重新执行 `sync_blocklist`。

### 2.2 会话免打扰

会话免打扰独立于 `conversation` 表：

```sql
CREATE TABLE IF NOT EXISTS mutelist (
    uid        INTEGER NOT NULL,
    to_uid     INTEGER NOT NULL DEFAULT 0,
    group_id   INTEGER NOT NULL DEFAULT 0,
    status     INTEGER NOT NULL DEFAULT 0,
    seq        INTEGER NOT NULL DEFAULT 0,
    updated_at INTEGER NOT NULL,
    PRIMARY KEY (uid, to_uid, group_id)
);
CREATE INDEX IF NOT EXISTS idx_mutelist_seq ON mutelist(uid, seq);

CREATE TABLE IF NOT EXISTS mutelist_version (
  uid         INTEGER PRIMARY KEY,
  gc_safe_seq INTEGER NOT NULL DEFAULT 0,
  max_seq     INTEGER NOT NULL DEFAULT 0
);
```

`to_uid` 与 `group_id` 二选一。`status=1` 表示开启免打扰，`status=0xff` 表示关闭 tombstone，方便其他设备关闭静音状态；后台 Mute GC 会按批清理关闭记录并推进 `mutelist_version.gc_safe_seq`，持久存储过旧游标收到 `seq_too_old` 后清空本地免打扰表和游标，再从 `last_seq=0, rebuild=true` 重新执行 `sync_mutelist`。

### 2.3 会话未读

未读数存储在 `conversation.unread_count`，由 `ConversationStore.Upsert` 的 `ConversationUnreadMode` 控制：

| 模式 | 场景 | unread 行为 |
|---|---|---|
| `ConversationUnreadIncrement` | 非静音会话收到他人新消息 | 插入为 1，更新为 `unread_count + 1` |
| `ConversationUnreadKeep` | 自己发送、撤回最终态、静音会话收到消息 | 插入为 0，更新保持原值 |
| `ConversationUnreadReset` | 显式 `clear_unread` | 清零 |

前端不维护完整未读 Map；会话列表角标读取分页项 `unread_count`，聊天 Tab 红点读取 `get_unread_count`。

## 3. Action 与通知

| 能力 | Action | 说明 | 通知 |
|---|---|---|---|
| 屏蔽列表 | `block_user` | 屏蔽指定用户，清理本侧待处理好友请求快照 | `blocklist:updated`，必要时追加 `contacts:updated` |
| 屏蔽列表 | `unblock_user` | 解除屏蔽 | `blocklist:updated` |
| 屏蔽列表 | `get_blocklist` | 分页读取当前生效屏蔽列表，支持 `uid` / `uids` 单个或多个过滤 | 无 |
| 屏蔽列表 | `sync_blocklist` | 按 `last_seq` 增量同步，包含 tombstone；过旧游标返回 `seq_too_old` | 无 |
| 免打扰 | `mute_conversation` | 开启单聊 / 群聊会话静音状态 | `conversations:mutelist-updated` |
| 免打扰 | `unmute_conversation` | 关闭单聊 / 群聊会话静音状态 | `conversations:mutelist-updated` |
| 免打扰 | `get_mutelist` | 管理页按分页读取当前静音会话，支持 `to_uid` / `group_id` 或批量过滤 | 无 |
| 免打扰 | `sync_mutelist` | 按 `last_seq` 增量同步，包含 `status=0xff` tombstone；过旧游标返回 `seq_too_old` | 无 |
| 红点 | `get_unread_count` | 返回所有正常会话未读数之和 | 无 |
| 红点 | `clear_unread` | 清零指定会话未读（只清未读不动 seq；会话行已 GC 时不重建，等下一条消息到来自然追平） | `conversations:clearunread` |
| 红点 | `get_contact_count(status=CONTACT_STATUS_PENDING_INCOMING)` | 获取待我处理的好友请求总数（不含自己发出、等对方处理的 `CONTACT_STATUS_PENDING_OUTGOING`） | 无 |

客户端收到 `blocklist:updated` / `conversations:mutelist-updated` 后只失效当前可见状态，需要展示时调用过滤分页或增量接口追平；启动阶段不全量拉取屏蔽列表或免打扰集合。分页读取、增量同步、tombstone 和 instant / 持久存储行为统一遵守 [`docs/architecture/同步机制方案.md`](../../docs/architecture/同步机制方案.md)；本文只补充屏蔽列表、免打扰和红点的业务约束。

Blocklist / Mutelist GC 只清理非生效记录，不删除当前生效的屏蔽或静音状态；清理后会推进各自 `*_version.gc_safe_seq`，与通讯录一样要求过旧游标重建当前状态。

## 4. 行为规则

### 4.1 屏蔽列表

| 场景 | 任一方向存在屏蔽列表时 |
|---|---|
| `add_friend` | 拒绝，避免继续建立主动社交链路 |
| `accept_friend` | 任一方向存在屏蔽列表时拒绝 |
| `send_message` 单聊 | 拒绝，返回模糊错误文案，避免泄露具体屏蔽方向 |
| `send_message` 群聊 | 不拦截，V1 不做群级内容屏蔽 |
| `search_user` | 返回空结果 |

屏蔽不自动删除好友、不删除历史消息、不清空会话、不清零未读。屏蔽与撤回互相独立。

### 4.2 会话免打扰

静音会话仍然：

- 写入消息收件箱。
- 更新会话最新预览和排序。
- 推送 `messages:received` 信号。

静音期间收到的新消息不会：

- 递增 `unread_count`。
- 为 `get_unread_count`、聊天 Tab 红点和未读摘要增加新的未读数。

如果会话在开启静音前已经有未读，当前实现不会在设置静音时自动清零，这部分既有 `unread_count` 仍会参与 `get_unread_count`，直到用户打开会话或显式调用 `clear_unread`。

打开静音会话时仍可调用 `clear_unread`，该操作保持幂等。

### 4.3 红点

| 位置 | 数据来源 | 清除条件 |
|---|---|---|
| 会话列表头像角标 | 当前 `LocalConversation.unreadCount` | 打开会话并调用 `clear_unread` |
| 导航栏聊天 Tab | `getUnreadCount() > 0` | 所有会话未读清零；静音前已有未读也需要 `clear_unread` 清掉 |
| 导航栏通讯录 Tab | `get_contact_count(status=CONTACT_STATUS_PENDING_INCOMING)` 或本地 PENDING_INCOMING 联系人索引 | 所有待我处理的好友请求被接受或拒绝 |
| Requests Tab（"请求"标签本身） | 同上 | 同上 |

普通 `messages:received` 只触发消息 / 会话分页同步，不在前端本地自行 `+1`，避免与服务端免打扰判定不一致。

## 5. 前端与 SDK 约束

- SDK / UIKit 启动会初始化 DataGateway 并建立会话分页游标；instant 模式不常驻完整屏蔽列表 / 免打扰集合，持久存储模式会把屏蔽列表和免打扰同步为本地持久存储副本，但公开读取仍走过滤分页。
- 屏蔽列表状态只由 `blockUser` / `unblockUser` 本地操作和 `getBlocklist` 过滤分页读取写入；`sync_blocklist` 仅供 SDK 数据层内部增量同步。
- 免打扰状态只由 `muteConversation` / `unmuteConversation` 本地操作和 `getMutelist` 过滤分页读取写入；`sync_mutelist` 仅供 SDK 数据层内部增量同步。
- 会话列表静音图标可按当前页需要单点查询或使用管理页缓存，不要求 SDK 常驻完整集合。
- UI 不应根据错误文案判断“被谁屏蔽”，发送失败只展示通用提示。
