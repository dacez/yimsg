## 1. 目标与原则

Yimsg SDK 长期运行时内存必须严格可控，因此所有长期驻留集合都建立在固定容量的有界结构之上：

- 所有长期驻留集合必须有硬上限，禁止无界 Map / Set / Queue 长期增长。
- 所有集合容量必须静态可估算，SDK 具备「理论峰值内存可估算」能力。
- 网络协议整包最大仍为 64KB（见协议帧约束），与有界集合容量共同构成内存上界。
- 结构必须易于跨语言（Rust / Go / C）复刻：开放寻址 + 固定桶 + 桶内线性扫描，无链表、无动态 chaining、无堆碎片，优先 cache locality 与可预测内存。

### 1.1 Message 不长期驻留

SDK 内存中**不允许**保存 `msg_id -> Message` 的长期全局缓存 Map。Message 属于「窗口态数据」：当前可见窗口、preload window、正在发送、短期 dedup。长期消息存储依赖持久层（SQLite / IndexedDB / OPFS）。增量同步单批最多 `DEFAULT_SYNC_BATCH_SIZE`（200）条，派发后立即释放，不累积。

## 2. BoundedU64Map\<V\>

真正固定容量的 `uint64 -> V` 映射，是有界集合基础设施的核心。

### 2.1 uint64 key

TypeScript 不依赖 string key map。uint64 用 `(hi, lo)` 两个 uint32 表示（`keysHi` / `keysLo`），可无损表达超过 `2^53` 的雪花式 id：

```
value = (BigInt(hi) << 32) | BigInt(lo)
```

`u64.ts` 提供 `parseU64` / `u64ToString` / `hashU64` / `u64Equal` / `nextPow2`。禁止 `"u:123"` 字符串 key 与无界 object key map。

### 2.2 布局

```
keys:   keysHi / keysLo  Uint32Array[bucketCount * bucketCapacity]
states:                  Uint8Array  [bucketCount * bucketCapacity]   // 0=EMPTY 1=OCCUPIED
seqs:                    Float64Array[bucketCount * bucketCapacity]   // FIFO/LRU 顺序，reject 策略不分配
values:                  Array<V>    [bucketCount * bucketCapacity]
```

- bucket 选择：`hashU64(hi, lo) & (bucketCount - 1)`，`bucketCount` 必须是 2 的幂。
- `bucketCapacity` 默认 8，bucket 内线性扫描（无探测溢出到其它 bucket）。
- 容量向上对齐到 `bucketCount(2^n) * bucketCapacity`，运行期固定不增长（`computeBoundedCapacity`）。
- 后续 Rust/Go/C 版本可每 bucket 一把锁；TypeScript 单线程不需要锁。

### 2.3 淘汰策略

| 策略 | 行为 | 用途 |
|------|------|------|
| `reject` | bucket 满时拒绝新 key，`rejectCount++`，返回 `false` | pending request（不允许新请求淘汰已有请求） |
| `fifo` | bucket 满时淘汰本桶最早插入的 key | cache 默认 |
| `lru` | 访问刷新顺序，淘汰本桶最久未访问的 key | 可选 |

## 3. BoundedU64Set

与 `BoundedU64Map` 共用开放寻址 + 固定桶布局，但不保存 value，专用于「待拉取 / 在飞」这类 uint64 去重队列。默认 reject 策略：容量满时 `add()` 返回 `false`，不允许新 key 淘汰已有 key。提供 `drain()` 原子排空。

## 4. BoundedQueue\<V\>

固定容量环形缓冲 FIFO 队列：`enqueue` / `dequeue` / `peek` / `clear` / `size` / `capacity`。两种溢出策略：

- `reject`：队满时 `enqueue()` 返回 `false`，保留旧元素（默认）。
- `overwrite_oldest`：队满时丢弃队首最旧元素再写入。

## 5. 运行时统计

每个有界集合都暴露 `stats(): BoundedStats`：`size`、`capacity`、`bucketCount`、`bucketCapacity`、`rejectCount`、`evictionCount`、`loadFactor`（队列以 `bucketCount=1`、`bucketCapacity=capacity` 统一语义暴露）。`YimsgClient.getBoundedCollectionStats()` 聚合显示信息缓存与待响应请求的统计，用于 benchmark / debug / 回归监控。

## 6. 第一阶段承载的集合

| 集合 | 承载结构 | 策略 | 说明 |
|------|---------|------|------|
| `pendingRequests`（WsTransport） | `BoundedU64Map<PendingReq>` | reject | key = request_id；`sendBinary()` 在 `size ≥ maxPendingRequests` 时先拒绝，底层 reject 进一步保证 `size ≤ capacity` |
| 用户显示信息缓存 | `BoundedU64Map<DisplayCacheEntry>` | fifo | key = uid |
| 群显示信息缓存 | `BoundedU64Map<DisplayCacheEntry>` | fifo | key = group_id；与用户缓存完全独立 |
| 显示信息待拉取 / 在飞队列 | `BoundedU64Set`（用户/群各 2 个） | reject | 满则抛 `ValidationError`，用户与群上限独立 |

用户与群缓存严格拆分（不使用 `kind + id` 统一 map），允许少量容量浪费，换取极简结构、纯 uint64 key、无 uid/group_id 冲突与跨语言一致性。

**未纳入**（明确禁止）：全局 Message Map、无限 Message Cache。

## 7. 峰值内存估算

`YimsgClient.estimateMaxMemoryBytes(options)` 把所有有界集合纳入理论峰值估算。每个有界集合满载字节由 `estimateBoundedU64MapBytes` / `estimateBoundedU64SetBytes` / `estimateBoundedQueueBytes` 计算（含固定 slot 结构开销 + 值/引用字节）：

```text
totalBytes
  = profileUserCacheBytes   // 用户显示信息 BoundedU64Map(fifo) 满载
  + profileGroupCacheBytes  // 群显示信息 BoundedU64Map(fifo) 满载
  + profileQueueBytes       // 4 × 待拉取/在飞 BoundedU64Set 满载
  + pendingRequestsBytes    // 待响应请求 BoundedU64Map(reject) 满载
  + syncBatchBytes          // DEFAULT_SYNC_BATCH_SIZE × BYTES_PER_SYNC_MESSAGE（常量）
  + forwardBundleBytes      // DEFAULT_MAX_FORWARD_BUNDLE_BYTES（常量）
  + baselineBytes           // SDK_BASELINE_BYTES（常量）
```

容量参数集中在 `sdk-defaults.ts`（`DISPLAY_CACHE_BUCKET_CAPACITY`、`DISPLAY_QUEUE_LOAD_FACTOR`、`PENDING_REQUEST_BUCKET_CAPACITY`、`PENDING_REQUEST_LOAD_FACTOR`、`BYTES_PER_DISPLAY_CACHE_VALUE`、`BYTES_PER_PENDING_REQUEST_VALUE` 等）。
