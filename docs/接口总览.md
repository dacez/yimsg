## 1. 规范

### 1.1 分层边界

Yimsg 的公开调用面分三层：

| 层级 | 入口 | 主要调用者 | 职责 |
|---|---|---|---|
| UIKit | `mount()`、`MountHandle`、widget 事件 | 第三方页面、项目主应用、九宫格控制台 | 提供完整聊天 UI，隐藏业务同步和 DOM 细节 |
| SDK | `YimsgClient` | 自定义 UI、UIKit、自动化场景 | 管理 WebSocket、认证、会话、分页读取、同步、缓存和事件 |
| 服务端 | WebSocket protobuf Type、`POST /api/upload`、`GET /media/*`、静态资源 | SDK DataGateway / transport、浏览器或宿主页面 | 执行业务校验、写入分片存储、返回 protobuf 定义的响应、发送轻量通知和提供资源访问 |

UIKit 不直接访问服务端 action；UIKit 只使用 `YimsgClient`。业务方自定义 UI 时，也应优先使用 SDK 门面，而不是自己拼 WebSocket 包。

### 1.2 设计理念

| 原则 | 说明 |
|---|---|
| WebSocket protobuf 帧为主接口 | 注册、登录、消息、联系人、群组、会话偏好都走 WebSocket interface；HTTP 只用于上传、媒体访问和静态资源 |
| protobuf 单一事实源 | `internal/protocol/yimsg.proto` 定义 type、request、response、BaseRequest、BaseResponse、ErrorCode 和 notification |
| 轻通知 + 主动拉取 | 服务端通知只表达“有变化”；客户端收到后用分页读取、游标同步或单点查询追平 |
| 分页优先 | 会话、消息、联系人、屏蔽列表、免打扰和群成员都按分页读取，避免无界集合进入 JS 堆 |
| SDK 统一同步 | memory 与 persistent 模式差异由 SDK DataGateway 屏蔽，UI 只消费 SDK 的只读事件和分页方法 |
| 应用层聚合 | 服务端按 `uid`、`username`、`group_id`、`token` 四类路由键分片；跨表、跨分片聚合在应用层完成，避免 JOIN |
| 鉴权先注册在线态 | `login` / `authenticate` 成功时，服务端先把连接注册到在线表，再返回成功响应，避免紧随其后的推送丢失 |

### 1.3 命名规范

**函数 / 方法：**

| 层 | 命名风格 | 示例 |
|---|---|---|
| UIKit 挂载函数 | camelCase | `mount`、`openConversation`、`setTheme` |
| SDK 公开方法 | camelCase | `sendMessage`、`getConversations`、`favoriteGroup` |
| SDK 事件名 | domain:event | `messages:received`、`contacts:updated` |
| SDK DataGateway 读取方法 | snake_case， 对齐 action； 同步 action 只作为持久存储内部实现， 不进入 DataGateway 接口 | `get_messages`、`get_contacts` |
| WebSocket action | snake_case，动词 + 对象；由 protobuf `Type` 注释映射到 type | `send_message`、`update_group_info` |
| Go Service 导出方法 | PascalCase | `SendMessage`、`ListByConversation` |
| Go 局部变量 / 参数 | camelCase | `friendUID`、`groupID`、`lastSeq` |

**字段：**

| 场景 | 规则 | 示例 |
|---|---|---|
| SDK 入参 / 返回字段 | camelCase | `toUid`、`groupId`、`cursor` |
| WebSocket protobuf 字段 | snake_case；protobuf wire 字段 | `target`、`group_id`、`page` |
| Go 协议结构字段 | PascalCase + JSON tag | `ToUID json.Number \`json:"to_uid,omitempty"\`` |
| 数据库列名 | snake_case | `friend_uid`、`sort_key`、`search_text` |
| 批量字段 | 复数 | `uids`、`group_ids`、`member_uids` |
| 同步游标字段（`sync_*`） | 统一词表 | `last_seq`、`limit`、`rebuild` |
| 展示分页字段（`get_*`） | 不透明 keyset 游标 | `page.cursor`、`page.direction`、`page.limit`、`page.around` |

**action 命名细则：**

- 读：`get_*` / `sync_*`
- 写：`send_*` / `mark_*` / `add_*` / `remove_*` / `update_*` / `block_*` / `unblock_*`
- 会话目标统一使用 `ConversationTarget`：私聊为 `{ uid }`，群聊为 `{ group_id }`。

新增接口必须显式完成 `camelCase ↔ snake_case` 映射，禁止跨层透传未转换字段。

### 1.4 协议与认证约束

**帧与请求 / 响应基础字段：**

- WebSocket 帧格式固定为 `magic:uint8('M') + codec:uint8(bitfield) + reserved:uint8(0) + checksum:uint8(CRC-8) + size:uint16 + request_id:uint64 + type:uint16 + body`。
- 协议整包上限是 `0xffff` 字节；header 为 16 字节，所以 `size` 最大是 `65519`。超限时返回 `FRAME_TOO_LARGE` 或拒绝发送。
- `magic` 用于识别 Yimsg 帧；`reserved` 当前固定为 0；`checksum` 是将 checksum 字节置 0 后对完整 frame 计算的 CRC-8（poly `0x07`、init `0x00`），用于发现截断、错位和常见传输损坏。
- `codec` 是位域：bit0 表示大小端（0=big-endian，1=little-endian），bit1-4 是 version（当前 1），bit5-7 保留为 0，并与后续 `reserved` 字节连续。
- `type=0` 是无效值；除 `ErrorCode` 的 `0=OK` 外，protobuf enum 的 `0` 都只表示 invalid / reserved / error。
- protobuf 是唯一协议源。body 只保留 protobuf wire format（`codec='b'`）。
- 每个接口有独立 Request / Response。`uid` 与 `request_id` 从帧头解析后由服务端框架填入 `BaseInfo` 结构体传入业务方法，proto 中不再包含 `BaseRequest`。Response 使用 `BaseResponse base = 1` 承载 `ErrorCode code` 和 `msg`。
- 业务字段从 protobuf 字段号 10 开始，不能在每个响应里重复定义 `code` / `msg`。

协议单一事实源、代码生成范围和契约测试策略见 [`protocol/README.md`](protocol/README.md)；type / 错误码 / 通知速查表见 [`generated/协议接口表.md`](generated/协议接口表.md)。本文继续维护当前完整接口矩阵和 SDK ↔ 服务端映射。

**认证：**

- 免认证 action 只有 `register`、`login`、`authenticate`；其余 action 都要求连接已认证。
- 当前 `uid` 由服务端连接状态写入 `BaseInfo`，客户端无需也不应传自己的 `uid`。
- `login` / `authenticate` 成功时，服务端先完成在线注册，再发送成功响应，并在响应发出后启动该连接的推送转发。

**数值编码：**

`uid`、`friend_uid`、`to_uid`、`group_id` 等 int64 字段在 JSON 中按字符串传输；SDK 对外 ID 类型同样使用字符串，避免前端数字精度丢失。

`msg_id` 是原生 string 类型（不是 int64）：UUIDv7 的 base64url 编码，固定 22 字符。整个项目（protobuf / Go / TypeScript / SQLite / JSON / WebSocket）msg_id 一律 string，禁止任何二进制 UUID 表示。用户消息的 msg_id 只由 TypeScript SDK 生成并随 `send_message` 上送，服务端只做校验、保存、回传与幂等；缺失或非法直接拒绝；系统消息由服务端发起、无 SDK 来源，是唯一允许由服务端生成 msg_id 的消息类型。消息顺序只依赖 `seq` / `send_time`，不得以 `msg_id` 排序。

**分页与同步：**

- `limit` 默认 200，并受服务端 `[client].batch_max_limit` 裁剪；硬上限为 500。
- `get_*` 用于当前页 / 管理页读取；`sync_*` 用于 DataGateway 内部按游标追平变化；UI 不应把 `sync_*` 当成全量列表接口。
- tombstone、GC 水位和 `seq_too_old` 的恢复规则统一见 [`同步机制方案.md`](同步机制方案.md)。

### 1.5 `client_config`

`login` / `authenticate` 成功响应会返回 `client_config`：

| 字段 | 默认值 | 说明 |
|---|---:|---|
| `cache_ttl_seconds` | `604800` | 客户端显示信息缓存 TTL，默认 7 天 |
| `cache_max_entries` | `10000` | 客户端显示信息缓存最大条目数，用户和群合计计算 |
| `recall_window_seconds` | `120` | 撤回时限；`< 0` 表示显式禁用 |
| `batch_max_limit` | `500` | 服务端分页 / 批量接口和 SDK 单批网络请求上限，硬封顶 500 |

SDK 中对应 `ClientConfig` 字段为 `cacheTtlSeconds`、`cacheMaxEntries`、`recallWindowsSeconds` 和 `batchMaxLimit`；显示信息缓存 TTL / 最大条目数只使用构造参数或 SDK 默认值，认证后不运行期改变；`recallWindowsSeconds` 以后端返回为准，`batchMaxLimit` 同时存在前端构造参数和后端 `client_config.batch_max_limit` 时取较小值。调用方可通过 `getClientConfig()` 读取当前生效配置。

## 2. UIKit 接口简介

UIKit 是完整聊天 UI 的嵌入层，当前只发布 ESM 产物 `web/uikit/yimsg-uikit.js`。它通过 Shadow DOM 隔离样式，内部复用项目主应用的同一套视图。

| 接口 / 类型 | 功能 |
|---|---|
| `mount(target, options?)` | 把完整聊天 UI 挂载到宿主容器，返回 `MountHandle` |
| `MountTarget` | `HTMLElement` 或 CSS 选择器字符串 |
| `MountOptions` | 连接、认证、布局、存储模式、主题、语言和宿主回调 |
| `MountHandle.client` | 当前 UIKit 使用的 `YimsgClient` |
| `MountHandle.shadowRoot` | 当前实例的 ShadowRoot，便于调试和测试 |
| `MountHandle.unmount()` | 卸载 UIKit，解绑事件，清空 Shadow DOM；自建 client 时会登出并销毁 |
| `MountHandle.setTheme(theme)` | 运行期切换主题 |
| `MountHandle.setLocale(locale, messages?)` | 运行期切换语言和文案覆盖 |
| `MountHandle.openConversation(target)` | 程序化打开直聊或群聊会话 |
| `MountHandle.logout()` | 程序化登出并回到认证页 |
| `MountHandle.on(event, handler)` | 监听 widget 级事件，返回解绑函数 |
| `WidgetEvents` | `authenticated`、`logout`、`message`、`conversation:open`、`error` |
| `UIKitMode` | `memory`、`persistent`、`persistent-cleardata` |
| `ThemeOption` / `LocaleOption` | 主题和语言配置 |
| `YimsgClient` | UIKit 包重导出 SDK 门面，方便宿主复用已认证 client |

宿主容器硬性最小尺寸为 `320 x 360 px`；低于该尺寸时显示“容器太小”提示。推荐业务容器至少 `360 x 420 px`。

## 4. UIKit 接口详解

### 4.1 `mount(target, options?)`

```ts
function mount(target: HTMLElement | string, options?: MountOptions): MountHandle;
```

`mount()` 会解析宿主容器，创建或复用 `shadowRoot`，注入完整 UIKit 骨架和样式，创建独立 `AppInstance`，绑定主题、语言、存储和事件桥接，最后返回运行期控制句柄。

| 参数 | 类型 | 说明 |
|---|---|---|
| `target` | `HTMLElement \| string` | 宿主元素或 CSS 选择器；选择器必须命中 `HTMLElement` |
| `options` | `MountOptions` | 可选挂载配置，见下节 |

返回值是 `MountHandle`。同一页面可多次调用 `mount()` 创建多个互相隔离的实例；`instanceId` 用于区分持久化本地库和运行态状态。

### 4.2 `MountOptions`

| 字段 | 类型 | 说明 |
|---|---|---|
| `wsUrl` | `string` | WebSocket 服务地址，透传给自建 `YimsgClient` |
| `uploadUrl` | `string` | HTTP 上传地址，默认 `/api/upload` |
| `requestTimeout` | `number` | 请求超时毫秒数 |
| `reconnectInterval` | `number` | 自动重连间隔毫秒数 |
| `heartbeatInterval` | `number` | 心跳间隔毫秒数，`0` 表示禁用 |
| `recallWindowSeconds` | `number` | 保留字段，当前 `mount()` 自建 client 不读取也不透传 |
| `instanceId` | `string` | 实例 ID，用于本地状态和持久化库隔离 |
| `token` | `string` | 宿主已有 token，挂载后自动 `authenticate` |
| `getToken` | `() => string \| Promise<string>` | 异步 token 提供者，挂载时调用一次 |
| `client` | `YimsgClient` | 复用宿主已有 SDK client；若已 ready 则跳过登录页 |
| `layout` | `'desktop' \| 'mobile' \| 'auto'` | 强制布局；默认按容器宽度判断 |
| `mode` | `UIKitMode` | `memory`、`persistent` 或 `persistent-cleardata`，默认 `memory` |
| `theme` | `ThemeOption` | `light`、`dark`、`auto` 或主题 token 覆盖对象 |
| `locale` | `LocaleOption` | `zh-CN`、`en` 或 `auto` |
| `messages` | `Partial<Messages>` | 内置文案局部覆盖 |
| `onReady` | `(client) => void` | 挂载并完成事件绑定后触发；不代表已登录 |
| `onAuthenticated` | `({ token, uid, event }) => void` | 登录或 token 鉴权成功后触发 |
| `onLogout` | `() => void` | 主动登出或被踢下线后触发 |
| `onMessages` | `(messages) => void` | 新消息批次到达或本端发送后触发；一次合并的多条会一起回调 |
| `onConversationOpen` | `(descriptor) => void` | 用户在 UI 中打开会话后触发 |
| `onError` | `(error, context) => void` | 用户可见错误、模式降级或内部异步错误 |

`recallWindowSeconds` 保留在 `MountOptions` 中用于兼容配置形状；实际撤回时限以服务端 `client_config` 为准。若宿主需要在认证前覆盖 SDK 初始值，应自行创建 `YimsgClient` 并通过 `client` 传入。

`mode` 语义：

| mode | SDK 映射 | 行为 |
|---|---|---|
| `memory` | `startSession({ storage: 'memory' })` | 内存会话，刷新即丢失，所有环境可用 |
| `persistent` | `startSession({ storage: 'persistent' })` | 请求持久化会话；不可用时 SDK 降级为 memory 并通过错误回调提示 |
| `persistent-cleardata` | `startSession({ storage: 'persistent', resetLocalData: 'current-user' })` | 先清理当前用户当前实例本地数据，再启动持久化会话 |

### 4.3 `MountHandle`

| 字段 / 方法 | 参数 | 说明 |
|---|---|---|
| `client` | 无 | 当前 UIKit 使用的 `YimsgClient` |
| `shadowRoot` | 无 | 当前实例 ShadowRoot |
| `unmount()` | 无 | 销毁 UIKit、解绑事件、清空 Shadow DOM；自建 client 时会登出并销毁 |
| `setTheme(theme)` | `ThemeOption` | 运行期切换主题 |
| `setLocale(locale, messages?)` | `LocaleOption`、`Partial<Messages>` | 运行期切换语言和文案覆盖 |
| `openConversation(target)` | `{ friendUid?: string; groupId?: string }` | 打开直聊或群聊会话；直聊会构造本地会话描述 |
| `logout()` | 无 | 程序化登出并回到认证页 |
| `on(event, handler)` | `keyof WidgetEvents`、handler | 监听 widget 级事件，返回解绑函数 |

### 4.4 Widget 事件

| 事件 | 载荷 | 用法 |
|---|---|---|

## 5. SDK 接口详解

### 5.1 生命周期与配置

| 方法 | 签名 | 功能 / 用法 |
|---|---|---|
| `constructor` | `(options?: ClientOptions)` | 创建 SDK 实例。`wsUrl` 默认按当前页面协议推导为 `/ws`，`uploadUrl` 默认 `/api/upload` |

### 5.2 认证接口

| 方法 | 参数 | 返回 | 说明 |
|---|---|---|---|
| `register` | `username, password, nickname` | `Promise<void>` | 注册用户；成功后仍需登录 |

| `conversations:mutelist-updated` | `{ snapshot, reason }` | 重新读取免打扰列表或单点状态 |
| `display:updated` | `{ keys, scope }` | 重新读取用户 / 群显示缓存 |
| `error` | `{ error, context, snapshot }` | 统一错误上报和 UI 提示 |

事件方法来自 `EventEmitter`：`on`、`off`、`once`、`listenerCount`、`removeAllListeners`。

## 6. 服务端接口详解

服务端对外暴露三类接口：WebSocket protobuf Type（业务主接口）、HTTP 上传 / 静态资源、推送通知（单向下行）。WebSocket frame header 承载 `request_id` 和 `type`；body 使用 protobuf 定义的 request / response，失败状态在 `BaseResponse.code/msg` 中。逐 type 的 request / response 类型与字段以协议表 [`generated/协议接口表.md`](generated/协议接口表.md) 为准，本节仅保留 SDK 映射速查。

### 6.3 推送通知矩阵

服务端推送是单向下行的轻通知，只表达“有变化”。memory 模式由客户端再走分页或带过滤参数的 `get_*` action 读取服务端；持久存储模式由客户端走同步 action 追平本地副本。

| 服务端动作 | 通知类型 | 接收方 | 客户端处理 |
|---|---|---|---|
| `send_message` 私聊 | `messages:received`（携带 `msg_id`） | 发送者和接收者全部在线连接 | SDK 累积各通知 `msg_id` 批量拉内容供 `onMessages`，persistent 先同步本地；UI 收到信号用 `get_conversations`/`get_messages` 重绘 |
| `send_message` 群聊 | `messages:received`（携带 `msg_id`） | 群成员和发送者多端 | 异步扇出完成后通知客户端，通知携带该批 `msg_id`；UI 按信号重绘 |
| `send_message`(`MESSAGE_TYPE_RECALL`) | `messages:received`（携带 recall 事件 `msg_id`） | 受影响会话成员 | 撤回走 send_message + RecallBody；UI 按信号重读 `get_messages`，撤回占位与事件按服务端最新态反映 |
| `delete_message` | `messages:delete`（携带 `msg_id`） | 操作者其他在线连接 | 命中数据窗口就地删消息（不拉取），持久存储删本地行；本端发起也走同一路径 |
| `delete_conversation` | `conversations:delete` | 操作者其他在线连接 | 命中数据窗口就地删会话（不拉取），持久存储删本地行；本端发起也走同一路径 |
| `add_friend` | `contacts:updated` | 被申请方 | 刷新待处理请求 |
| `accept_friend` / `reject_friend` | `contacts:updated` | 关系双方 | 刷新联系人或请求列表 |
| `delete_friend` | `contacts:updated` | 调用方在线连接 | 单向刷新通讯录或群收藏 |
| `get_user_infos` / `get_group_infos` | `contacts:updated` | 调用方在线连接 | 当联系人 / 群收藏的 `sort_key` / `search_text` 投影因昵称或群名变化而刷新时发送 |
| `block_user` / `unblock_user` | `blocklist:updated` | 操作者在线连接 | 刷新屏蔽列表状态 |
| `mute_conversation` / `unmute_conversation` | `conversations:mutelist-updated` | 操作者在线连接 | 刷新会话免打扰状态 |
| `logout` / 踢下线 | `session:kicked` | 被失效 token 对应连接 | 进入未登录态 |
| `clear_unread` | `conversations:clearunread` | 同 uid 其他在线连接 | 多端就地清未读红点（不拉取）；服务端只清未读不动 seq，会话行已 GC 时不重建，等下一条消息到来自然追平 |

### 6.4 特殊同步语义

- 撤回（`send_message` + `MESSAGE_TYPE_RECALL` + `RecallBody`）：服务端校验后覆盖原消息为撤回占位并写入一条 recall 事件消息；`get_messages` 和 `sync_messages` 返回 `MESSAGE_TYPE_RECALL` 消息，SDK 折叠为原消息占位更新，不把 event 暴露成额外可见消息。
- 消息 / 会话删除：`sync_messages`、`sync_conversations` 返回 `status=0xff` tombstone；持久 SDK 收到后删除本地 `messages` / `conversations` 行并推进游标。Message GC 和 Conversation GC 不推进 `messages_version.gc_safe_seq`，不会返回 `seq_too_old`，历史缺口允许漏同步。
- 免打扰关闭：`sync_mutelist` 返回 `status=0xff` tombstone；持久 SDK 收到后删除本地 `mutelist` 行。Mute GC 清理关闭 tombstone 后推进 `mutelist_version.gc_safe_seq`，过旧游标按重建流程处理。
- `get_contacts`：用于联系人分页读取，不是启动阶段全量快照接口。
- `get_blocklist` / `get_mutelist`：用于管理页分页读取，也用于详情页和当前可见对象的单个 / 多个过滤状态查询，不应在启动阶段全量拉取。
- `sync_contacts` / `sync_blocklist` / `sync_mutelist` / `sync_conversations`：仅持久存储 DataGateway 用于本地副本维护；其中通讯录、屏蔽列表、免打扰按 [`同步机制方案.md`](同步机制方案.md) 的过旧游标规则重建，消息和会话按消息式 `seq` 窗口同步且不使用 `seq_too_old`；memory 模式不调用 `sync_*`。

## 附录：新增接口落地清单

新增或修改接口时按以下顺序同步：

1. 明确命名：SDK 使用 camelCase，action 和协议字段使用 snake_case。
2. 更新协议：先改 `internal/protocol/yimsg.proto`，确保独立 Request / Response、`BaseRequest base = 1`、`BaseResponse base = 1` 和业务字段号从 10 开始，再运行 `go run ./tools/cmd/protocolgen`。
3. 更新服务端：`internal/ws/connection.go` frame 解码 / action 路由、`internal/service/` 业务和必要的存储读写实现。
4. 更新 SDK：`frontend/src/sdk/client.ts`、`frontend/src/sdk/modules/`、DataGateway 或缓存层。
5. 更新 UIKit：若能力暴露到 UI 或宿主回调，更新 `frontend/src/uikit/` 和相关视图。
6. 更新文档：本文、[`frontend/sdk接口说明.md`](frontend/sdk接口说明.md)、[`frontend/UIKit方案.md`](frontend/UIKit方案.md)；若影响同步域，先更新 [`同步机制方案.md`](同步机制方案.md)。
7. 更新测试：后端单元 / E2E、前端 unit / sdk / ui 按影响面补齐；实现代码变更后从仓库根目录运行 `./tools/run_all_tests.sh`。


### 分页语义说明

- **展示通道（`get_*`）统一使用不透明 keyset 游标**，请求带 `PageQuery{cursor, direction, limit, around}`，响应带 `PageInfo{start_cursor, end_cursor, has_more_backward, has_more_forward, total}`。游标由服务端按各列表展示序的 keyset 字段编码（base64url），对客户端不透明、原样透传，抗实时重排与中途增删漂移，已彻底取代 `offset` 与 `before_seq/after_seq/around_seq`。
- `direction` 以屏幕方向为准：`PAGE_DIRECTION_FORWARD` 向尾/向下翻（追加，续翻用上一页 `end_cursor`），`PAGE_DIRECTION_BACKWARD` 向头/向上翻（前插，续翻用上一页 `start_cursor`）；空 `cursor` + 方向表示从该端起点开始。展示序：消息 旧→新（取最新页用空游标 + BACKWARD）；会话/屏蔽/免打扰 新→旧；联系人按 `sort_key`（pending 按 `seq` 倒序）；群成员 `role` 倒序、`uid` 升序。
- `page.around` 用于锚点居中定位（搜索结果、引用消息、通知/收藏定位）：messages 传被定位消息的 `msg_id`，contacts 传排序首字母；`around` 与 `cursor` 互斥。`get_messages` 的 `msg_ids` 批量读取与 `page` 互斥，不分页、`page` 信息为空。
- 总数：`PageInfo.total` 给 `get_group_members` 承载成员总数（其余可为 `-1`/未知）；通讯录展示总数仍用专用的 `get_contact_count`。
- **同步通道（`sync_*`）不变**，仍用 `last_seq/limit/rebuild`，响应统一带 `has_more` 和 `cursor_seq`。
- `ConversationTarget` 只表达会话/消息目标；通讯录相关目标使用 `ContactTarget`，单个查询通过 repeated 字段传 1 个元素。
