## 1. 范围与原则

| 能力 | 当前范围 | 原则 |
|---|---|---|
| 强类型正文 | 所有消息正文用 protobuf `MessageBody`（oneof）表达，不再用 JSON / Markdown / EXT 字符串信封 | `msg_type` 与 `body.kind` 必须一致，不一致返回 `ERROR_INVALID_ARGUMENT` |
| 文本 / Markdown | `MESSAGE_TYPE_TEXT` / `MESSAGE_TYPE_MARKDOWN` | 前端安全渲染，服务端按长度校验 |
| 图片 / 文件 | `MESSAGE_TYPE_IMAGE` / `MESSAGE_TYPE_FILE`，仅用 `media_id` 引用媒体 | 媒体只存 id，不在 body 里塞长地址 |
| 引用 | `MESSAGE_TYPE_QUOTE` + `QuoteBody` | 携带 `quote_msg_id`、预览和当前回复文本 |
| 转发 | `MESSAGE_TYPE_FORWARD` + `ForwardBody` | 仅携带被转发的 `msg_ids` 与标题，不复制明文 / 不加密上传 |
| 撤回 | `MESSAGE_TYPE_RECALL` + `RecallBody`，走 `send_message` | 服务端校验后覆盖原消息为占位，并追加事件消息驱动增量同步 |

所有能力继续复用 `send_message`、`get_messages`、`sync_messages` 主链路；研发阶段不做旧格式兼容。

## 2. 消息类型与 MessageBody

`internal/dal/types.go` 中当前消息类型（与 protobuf `MessageType` 一致）：

```go
const (
    MsgText     int8 = 1
    MsgImage    int8 = 2
    MsgSystem   int8 = 3
    MsgFile     int8 = 4
    MsgRecall   int8 = 5
    MsgQuote    int8 = 6
    MsgForward  int8 = 7
    MsgMarkdown int8 = 8
)
```

正文统一用 `MessageBody` oneof（`internal/protocol/yimsg.proto`）：

```proto
message MessageBody {
  oneof kind {
    TextBody text = 1;        // MESSAGE_TYPE_TEXT
    ImageBody image = 2;      // MESSAGE_TYPE_IMAGE
    SystemBody system = 3;    // MESSAGE_TYPE_SYSTEM
    FileBody file = 4;        // MESSAGE_TYPE_FILE
    RecallBody recall = 5;    // MESSAGE_TYPE_RECALL
    QuoteBody quote = 6;      // MESSAGE_TYPE_QUOTE
    ForwardBody forward = 7;  // MESSAGE_TYPE_FORWARD
    MarkdownBody markdown = 8;// MESSAGE_TYPE_MARKDOWN
  }
}
```

`messages` 表正文列为 `body BLOB NOT NULL`（protobuf 编码），元数据列（`uid/seq/msg_id/from_uid/to_uid/group_id/msg_type/send_time/status`）继续列化。`search_text` 是由 `body` 派生的搜索投影，不是真实内容来源；`MessageBody` 只是线协议和 body 存储格式，不作为长期内存对象模型。

校验规则：

- `body` 缺失或 oneof 未设置返回 `ERROR_INVALID_ARGUMENT`。
- `msg_type` 与 `body.kind` 不一致返回 `ERROR_INVALID_ARGUMENT`。
- 文本上限 4096 字符；Markdown 上限 20000 字节；转发 `msg_ids` 上限 20。

## 3. 引用与转发

### 3.1 引用

`MESSAGE_TYPE_QUOTE` + `QuoteBody`：

| 字段 | 说明 |
|---|---|
| `quote_msg_id` | 被引用消息 ID |
| `quote_preview` | 被引用消息预览，参与 `search_text` |
| `text` | 当前回复文本（`TextBody`），参与 `search_text` |

SDK facade 暴露 `sendQuotedTextMessage(target, { text, quoteMsgId, quotePreview? })`，内部组装 `QuoteBody`。UI 从 `body.quote` 渲染引用块。

### 3.2 转发

`MESSAGE_TYPE_FORWARD` + `ForwardBody`：

| 字段 | 说明 |
|---|---|
| `msg_ids` | 被转发消息 ID 列表（上限 20） |
| `title` | 转发标题 / 附言，参与 `search_text` |

转发只携带被转发消息的 id 与标题，不再复制明文、不加密上传明细。UI 展示“转发 N 条”摘要与标题。

## 4. 撤回语义

### 4.1 用户规则

- 只允许发送者撤回自己发送的消息。
- 默认撤回时限为 `[message].recall_window_seconds = 120`；负数表示关闭撤回能力。
- 系统消息、撤回事件消息、已撤回占位消息不可再次撤回。
- 撤回不回滚历史未读数，也不额外增加未读数。
- 撤回统一通过 `send_message` + `MESSAGE_TYPE_RECALL` + `RecallBody` 表达，没有独立的 recall action。

`RecallBody` 字段：`msg_id`（被撤回原消息 ID）、`operator_uid`、`recall_time`、`text`（展示文案）。其中 `operator_uid`、`recall_time`、`text` **由服务端设置并覆盖客户端传入值**：`operator_uid` 取当前登录用户、`recall_time` 取服务端当前时间、`text` 由服务端按角色生成。同步表只搬运数据、不含逻辑。

展示文案：发送者侧 `你撤回了一条消息`；单聊接收方 `对方撤回了一条消息`；群聊其他成员 `{昵称} 撤回了一条消息`。

### 4.2 原消息占位

被撤回的原消息覆盖为 `MESSAGE_TYPE_RECALL` 占位 body（`UpdateByMsgID`，不 bump seq），其 `RecallBody.msg_id` 等于自身 `msg_id`，原正文被服务端脱敏不可再读。历史拉取只返回占位后的原消息。

### 4.3 撤回事件

撤回时额外追加一条新的 `MESSAGE_TYPE_RECALL` 事件消息（新 `msg_id`、新 `seq`），其 `RecallBody.msg_id` 指向被撤回的原消息 ID。撤回事件消息的 `msg_id` 就是本次 `send_message` 请求顶层的 `msg_id`（由 SDK 生成），服务端不另行生成。该事件进入普通消息流和同步流，用于 `sync_messages(last_seq)` 驱动其他设备 / 离线设备折叠原消息。SDK 收到 recall 事件后，按 `RecallBody.msg_id` 把原消息折叠为撤回占位，不把事件暴露成额外可见气泡。

## 5. 服务端流程

### 5.1 发送结构化消息

`send_message` 校验 `msg_type` 与 `body` 后复用现有私聊、群聊扇出与会话更新流程：

0. 校验请求顶层 `msg_id`：必须是合法 UUIDv7 base64url（22 字符），缺失或非法直接拒绝（`INVALID_ARGUMENT`）。`msg_id` 由 SDK 生成，服务端不补、不替换；服务端按 `(uid, msg_id)` 做幂等：相同 `msg_id` 重复发送（含 WebSocket 重试、重连重发）不重复入库，返回已有 `seq` 与相同 `msg_id`。仅系统消息（由服务端发起、无 SDK 来源）由服务端生成 `msg_id`。
1. 校验 `body` 存在、oneof 已设置、`msg_type` 与 `body.kind` 一致。
2. 按类型校验子字段（文本 / Markdown 长度、媒体 `media_id`、引用 `quote_msg_id` 与文本、转发条数）。
3. 私聊校验联系人与屏蔽列表；群聊校验成员身份。
4. 由 `body` 编码 `body BLOB` 并派生 `search_text`，写入消息、更新会话、推送 `messages:received`。

### 5.2 单聊撤回

1. 通过 `uid + msg_id` 找发送者自己的消息副本。
2. 校验发送者、会话范围、撤回时限和消息类型。
3. 覆盖发送者和接收者原消息为撤回占位 body（`RecallBody`）。
4. 双方各插入一条撤回事件消息（`RecallBody`，新 `msg_id`）。
5. 使用 `ConversationUnreadKeep` 更新会话预览和排序。
6. 向双方在线设备推送 `messages:received`。

### 5.3 群聊撤回

群聊撤回在主流程中直接向异步任务队列（`internal/taskqueue`）投递一个 `recalled` 群消息任务，靠幂等收敛避免“先发后撤”竞态：

- 原始群消息和撤回是两个独立任务，不再共享“最终态”行，也不再需要 outbox 的 version 收敛 / `MarkRecalled`。
- 撤回任务对每个目标用户：原消息副本不存在则插入撤回占位，已存在则 `UpdateByMsgID` 覆盖为占位（body 为 `RecallBody` 的 protobuf bytes），并写入一条撤回事件消息。
- 收敛由 DAL 的 `Insert(OR IGNORE)` + `UpdateByMsgID` 幂等保证：send / recall 两个任务无论先后或并发执行，最终都收敛到“占位 + 撤回事件”。

任务在执行前先持久化（启用持久化时），即使服务进程崩溃重启也会按入队顺序重放未完成任务，不会把旧正文重新补发给群成员。

## 6. 客户端处理

| 场景 | SDK / UI 行为 |
|---|---|
| 文本 / Markdown | `describeMessage(message)` 从 `body.text` / `body.markdown` 派生文本与安全 HTML |
| 图片 / 文件 | 从 `body.image` / `body.file` 的 `media_id` 还原 `/media/{image\|file}/{media_id}` |
| 引用 | 展示 `body.quote` 引用块与回复文本 |
| 转发 | 展示“转发 N 条”摘要与 `body.forward.title` |
| 撤回入口 | 仅自己发送、非系统 / 非 `MESSAGE_TYPE_RECALL`、未超过撤回时限时展示 |
| 收到 recall 事件 | 按 `RecallBody.msg_id` 折叠内存 / 持久存储本地副本中的目标消息为占位，不暴露事件气泡 |
| 会话预览 | 统一使用 `describeMessage(message).text`，撤回占位展示 `RecallBody.text` |
| 本地搜索 | 持久存储用本地派生的 `search_text` 做 LIKE 搜索，规则与服务端一致 |

## 7. 接口与配置

| 项 | 说明 |
|---|---|
| `send_message` | 发送强类型 `MessageBody` 消息（文本 / 图片 / 文件 / Markdown / 引用 / 转发 / 撤回）；响应 `msg_id, seq` |
| 撤回 | `send_message` + `msg_type=MESSAGE_TYPE_RECALL` + `body.recall`（仅需 `msg_id`）；响应原 `msg_id` 与撤回事件 `seq` |
| `get_messages` | 按会话用展示通道 keyset 游标分页（`page{cursor,direction,limit,around}`，`around` 传 msg_id 居中定位）或按 `msg_ids` 批量拉取消息，包含 `MESSAGE_TYPE_RECALL` 事件；`msg_ids` 与 `page` 互斥、不分页、`page` 为空，按 `msg_ids` 取消息只认 `uid`、忽略 target，供 `messages:received` 收到后按累积的通知 `msg_id` 批量取内容；SDK 对外折叠成原消息占位更新 |
| `sync_messages` | 增量同步消息，包含 `MESSAGE_TYPE_RECALL` 事件；持久存储 DataGateway 折叠落地 |
| `[message].recall_window_seconds` | 撤回时限；默认 120，负数禁用 |
