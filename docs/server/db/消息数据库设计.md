## 1. 设计目标与约束

- 支持私聊和群聊消息的发送与存储
- 支持基于 seq 的增量同步（客户端维护 last_seq）
- 支持按会话翻页拉取历史消息
- 每用户消息数有上限，超限由 GC 清理
- 消息写入必须幂等（同一 msg_id 不重复写入）

## 2. 核心思想

### 扇出写模型

每条消息写入每个接收者的独立收件箱。一条群消息发给 M 个成员 = M 次 INSERT。

优势：
- 读路径极简：`WHERE uid = ? AND seq > ?`，单分片、走主键
- 删除互不影响：每个用户的 GC 独立运行
- 无跨分片读：同步消息永远只读自己的分片

代价：
- 写放大 M 倍（群消息）
- 存储放大 M 倍

### 原子 seq 分配

seq 在每个 uid 内单调递增，由 `messages_version.max_seq` 在写事务内原子分配：

```sql
UPDATE messages_version
SET max_seq = max_seq + 1
WHERE uid = ?
RETURNING max_seq;

INSERT INTO messages (uid, seq, msg_id, ...)
VALUES (?, ?, ?, ...)
ON CONFLICT(uid, msg_id) DO NOTHING;
```

消息写入和 seq 分配在同一事务内完成。`ON CONFLICT` 保证幂等，重复 `msg_id` 不消耗新 seq。

### 会话物化视图

`conversations` 表是 `messages` 的物化索引，记录每个会话的最新消息摘要。每次写消息时同步更新：

```sql
INSERT INTO conversations (...) VALUES (...)
ON CONFLICT(uid, to_uid, group_id) DO UPDATE SET ...
WHERE excluded.seq > conversations.seq
```

仅当新消息的 seq 大于已有的 `seq` 时才更新，保证乱序写入不会回退。

`unread_count` 在 `Upsert` 时根据 `ConversationUnreadMode` 参数决定行为：
- `ConversationUnreadReset`：INSERT / UPDATE 都把 `unread_count` 置 0，仅用于显式清未读语义。
- `ConversationUnreadIncrement`：INSERT `unread_count=1`，UPDATE `unread_count+1`。
- `ConversationUnreadKeep`：INSERT 默认为 0，UPDATE 保持原未读数；发送者自己的消息也使用该模式，不隐式清掉已有未读。

消息和会话共用 `messages_version.max_seq` 分配 uid 内统一 seq；会话表只保存最后一条消息的定位和未读数，不保存消息正文、发送者、类型或发送时间。`get_conversations` / `sync_conversations` 会按 `seq` 从同 uid 分片的 `messages` 表读取最后一条消息并拼装协议响应中的 `last_msg` 和 `last_seq`。`clear_unread` 接口调用 `ClearUnread`：会话行存在时只把 `unread_count` 清零、**不动 seq**（避免重排）；会话行已缺失（多为已被 GC）时**不重建**——红点由 `conversations:clearunread` 事件在前端数据窗口内就地清除，后台会话表等下一条消息到来时自然重建追平。随后向同一用户的在线设备推送 `conversations:clearunread` 通知，客户端只在数据窗口内就地清未读红点、不触发拉取。

消息、会话与其他同步域的轻通知、分页读取、游标增量和 持久存储本地副本规则统一见 [`../../同步机制方案.md`](../../同步机制方案.md)。

## 3. 表设计

### messages — 消息收件箱（uid 分片）

```sql
CREATE TABLE IF NOT EXISTS messages (
    uid       INTEGER NOT NULL,
    seq       INTEGER NOT NULL,
    msg_id    TEXT    NOT NULL,
    from_uid  INTEGER NOT NULL,
    to_uid    INTEGER NOT NULL DEFAULT 0,
    group_id  INTEGER NOT NULL DEFAULT 0,
    msg_type  INTEGER NOT NULL DEFAULT 0,
    content   TEXT    NOT NULL DEFAULT '',
    send_time INTEGER NOT NULL,
    status    INTEGER NOT NULL DEFAULT 0,
    PRIMARY KEY (uid, seq)
);
  CREATE UNIQUE INDEX IF NOT EXISTS idx_messages_uid_msgid ON messages(uid, msg_id);
```

| 字段 | 类型 | 说明 |
|------|------|------|
| uid | INTEGER | 收件箱所属用户 |
| seq | INTEGER | 用户内单调递增序列号 |
| msg_id | TEXT | 全局消息 ID，UUIDv7 的 base64url 编码（22 字符），由 SDK 生成，作为幂等去重键 |
| from_uid | INTEGER | 发送者 |
| to_uid | INTEGER | DM 接收者（群消息为 0） |
| group_id | INTEGER | 群 ID（DM 为 0） |
| msg_type | INTEGER | 1=文本, 2=图片, 3=系统, 4=文件, 5=Markdown, 6=通用扩展信封；0 保留为无效值 |
| content | TEXT | 消息内容 |
| send_time | INTEGER | 发送时间（ms） |
| status | INTEGER | 0=正常，0xff=已删除 tombstone；用于同步后删除 SDK 本地行 |

索引说明：
- **PK `(uid, seq)`**：增量同步 `WHERE seq > last_seq`；展示翻页按 keyset 游标 `WHERE seq </> ?`（游标编码 seq）
- **`idx_messages_uid_msgid`** UNIQUE：去重 `ON CONFLICT(uid, msg_id) DO NOTHING`

### messages_version — 消息 / 会话统一序列（uid 分片）

```sql
CREATE TABLE IF NOT EXISTS messages_version (
  uid         INTEGER PRIMARY KEY,
  gc_safe_seq INTEGER NOT NULL DEFAULT 0,
  max_seq     INTEGER NOT NULL DEFAULT 0
);
```

`messages_version.max_seq` 是 `messages.seq` 和 `conversations.seq` 的唯一分配来源；删除消息或删除会话也会先递增它，再把对应行写成 `status=0xff`。`gc_safe_seq` 保留为 0，Message GC 和 Conversation GC 都不推进它，因此消息 / 会话同步不会返回 `seq_too_old`。

### conversations — 会话列表（uid 分片）

```sql
CREATE TABLE IF NOT EXISTS conversations (
    uid            INTEGER NOT NULL,
    to_uid         INTEGER NOT NULL DEFAULT 0,
    group_id       INTEGER NOT NULL DEFAULT 0,
    seq           INTEGER NOT NULL,
    last_msg_id    INTEGER NOT NULL,
    unread_count   INTEGER NOT NULL DEFAULT 0,
    status         INTEGER NOT NULL DEFAULT 0,
    PRIMARY KEY (uid, to_uid, group_id)
);
CREATE INDEX IF NOT EXISTS idx_conversations_seq ON conversations(uid, seq);
```

| 字段 | 类型 | 说明 |
|------|------|------|
| uid | INTEGER | 所属用户 |
| to_uid | INTEGER | DM 对方（群聊为 0） |
| group_id | INTEGER | 群 ID（DM 为 0） |
| seq | INTEGER | 最近会话事件 seq，与 `messages.seq` 共用 `messages_version`；协议响应字段仍为 `last_seq` |
| last_msg_id | INTEGER | 最后一条消息的 msg_id |
| unread_count | INTEGER | 该会话的未读消息数；接收方写入时根据会话免打扰状态决定 `+1` 或保持原值，发送方写入保持原未读数；`clear_unread` 调用后清零 |
| status | INTEGER | 0=正常，0xff=已删除 tombstone；用于删除事件同步，GC 按 `seq` 窗口直接清理会话行 |

## 4. 对外接口逻辑

### send_message（私聊）

```
1. MessageStore.Insert(sender, msgID, ...)                    — sender 的 uid 分片，从 `messages_version` 取 seq
2. ConversationStore.Upsert(sender, ..., ConversationUnreadKeep)        — own message 不增加也不清空已有未读
3. MessageStore.Insert(receiver, msgID, ...)                  — receiver 的 uid 分片，从 `messages_version` 取 seq
4. ConversationStore.Upsert(receiver, ..., recipientUnreadMode) — 非静音 `unread_count+1`，静音保持原值
5. 推送 messages:received 通知给 receiver
```

4 次写入，最多跨 2 个 uid 分片。

### send_message（群聊）

```
同步部分：
1. GroupStore.IsMember(groupID, sender)                      — group 分片
2. GroupStore.ListAllMembers(groupID)                             — group 分片
3. MessageStore.Insert(sender, msgID, ...)                   — sender 的 uid 分片，从 `messages_version` 取 seq
4. ConversationStore.Upsert(sender, ..., ConversationUnreadKeep)       — own message 不增加也不清空已有未读
5. 返回 Response{msg_id, seq} 给客户端

异步扇出（投递到 `internal/taskqueue` 的 `group_message` 任务）：
6. 对每个其他成员：
  - MessageStore.Insert(member, msgID, ...)
  - ConversationStore.Upsert(member, ..., recipientUnreadMode) — 非静音 `unread_count+1`，静音保持原值
7. 推送 messages:received 通知给所有在线成员

扇出任务在主流程中直接 `submitTask` 投递，由后台 worker 执行；任务内部按目标 uid 的分片分组，每组一个 goroutine 并发写入。
```

### clear_unread — 清除会话未读

```
ConversationStore.ClearUnread(uid, toUID, groupID)
→ UPDATE conversations SET unread_count = 0
  WHERE uid = ? AND to_uid = ? AND group_id = ? AND status = 活跃   // 不动 seq（避免重排）
// 会话行已被 GC（未命中）则不重建：红点由 conversations:clearunread 事件就地清除，
// 后台等下一条消息到来时自然重建追平。
推送 conversations:clearunread 通知给该用户的在线设备
```

客户端在以下两种时机调用：
- 打开某个会话时
- 已在当前会话中收到该会话的新消息时（防止未读数误增）

### sync_messages — 增量同步

```
MessageStore.Sync(uid, lastSeq, limit)
→ SELECT ... WHERE uid = ? AND seq > ? ORDER BY seq ASC LIMIT ?
```

客户端维护 last_seq，每次拉取 seq > last_seq 的消息。`sync_messages` 返回正常消息和 `status=0xff` 的删除 tombstone；持久 SDK 收到删除 tombstone 后删除本地 `messages` 行并推进 `msg_seq`。Message GC 物理删除旧消息后不会推进 `gc_safe_seq`，长时间未同步客户端可能漏掉已被 GC 的消息，这是消息域的预期语义。

撤回场景会通过 `get_messages` / `sync_messages` 返回 recall event，SDK 将其归一化为原消息占位更新；跨端同步语义见 [`../../同步机制方案.md`](../../同步机制方案.md)。

### get_messages — 会话消息翻页

三种模式：
- **向上翻页**：`beforeSeq` 游标，`WHERE seq < beforeSeq ORDER BY seq DESC LIMIT ?`
- **向下翻页**：`afterSeq` 游标，`WHERE seq > afterSeq ORDER BY seq ASC LIMIT ?` 后再按 seq 降序返回，便于 SDK 保持同一响应契约
- **定位跳转**：`aroundSeq` 中心点，UNION 前后各取 limit/2 条

### get_conversations — 会话列表

```
ConversationStore.List(uid, newer, boundarySeq, limit)  // 展示通道 keyset 游标
→ SELECT ... WHERE uid = ? AND status = 0 ORDER BY seq DESC LIMIT ? OFFSET ?
→ ConversationStore.Count 拼装 total
→ MessageStore.GetBySeq(uid, seq) 拼装 last_msg
```

展示通道按不透明 keyset 游标读取会话分页（活跃→沉默）。`get_conversations` 返回当前页内每个会话的 `unread_count`；全局未读总数由 `get_unread_count` 单独读取，避免会话分页每次都做全表汇总。

请求带 `targets`（`ConversationTarget[]`，参考 `get_messages` 的 `msg_ids`）时，走 `ConversationStore.GetByTargets`：按目标精确读取这些会话仍活跃的当前状态、忽略分页，缺失/已删除目标不返回。用于「轻通知（`conversations:clearunread` / `conversations:delete` / `messages:delete`）后定向刷新单个/多个会话」，遵守轻通知 + 主动拉取原则。

### get_unread_count — 未读总数

```
ConversationStore.TotalUnreadCount(uid)
→ SELECT COALESCE(SUM(unread_count), 0) FROM conversations
  WHERE uid = ? AND status = 0
```

返回所有正常会话的未读数之和，供导航红点等全局状态使用。静音会话已有未读不会在开启静音时自动清零，仍会参与该总数，直到用户打开会话或显式调用 `clear_unread`。

## 5. 垃圾回收

### Message GC — 消息 seq 保留范围

```
默认间隔：3600s（若运行时配置值 <= 0，内部兜底为 300s）
默认保留：retainSeqWindow = 100,000 个消息 seq
流程：
1. MessageStore.ListPurgeable(retainSeqWindow, limit=500, afterUID)
  → SELECT uid FROM messages WHERE uid > ? GROUP BY uid HAVING MAX(seq)-MIN(seq) > ? LIMIT ?
2. 对每个超标 uid：
   MessageStore.Purge(uid, retainSeqWindow)
  → DELETE FROM messages WHERE uid = ? AND seq <= max(messages_version.max_seq)-retainSeqWindow
```

Purge 和 Insert 操作在 seq 范围上不重叠（Purge 删旧的，Insert 写新的），无竞争。Message GC 不推进 `messages_version.gc_safe_seq`，因此不会触发 `seq_too_old`。

### Conversation GC — 会话 seq 窗口清理

```
默认间隔：3600s（若运行时配置值 <= 0，内部兜底为 300s）
默认保留：retainSeqWindow = 10,000 个会话 seq
流程：
1. ConversationStore.ListPurgeable(retainSeqWindow, limit=500, afterUID)
  → SELECT uid FROM conversations WHERE uid > ? GROUP BY uid HAVING MAX(seq)-MIN(seq) > ? LIMIT ?
2. ConversationStore.Purge(uid, retainSeqWindow)
  → DELETE FROM conversations WHERE uid = ? AND seq <= max(seq)-retainSeqWindow
```

Conversation GC 与 Message GC 使用同类语义：只保留服务端当前 `seq` 窗口内的数据，共用 `messages_version`，不维护 `conversation_version`，也不触发 `seq_too_old`。长时间未同步的客户端可能错过已被 GC 的会话变化；持久存储只要求继续同步服务端仍保留的会话行。

## 6. 并发控制

- **seq 分配原子性**：消息和会话删除在 writer 事务内递增 `messages_version.max_seq`，SQLite 单写保证 uid 内 seq 单调
- **msg_id 去重**：`ON CONFLICT(uid, msg_id) DO NOTHING`，同一消息重复写入自动忽略
- **conversation 乱序更新**：`WHERE excluded.seq > conversation.seq`，只有更新的 seq 能覆盖
- **读写分离**：同步和翻页走 Reader，写消息走 Writer，WAL 模式下互不阻塞
- **扇出并发**：不同目标 uid 可能落在不同分片，按分片分组后并发写入，同一分片内串行

## 7. 扩容策略

- **uid 分片扩容**：messages、messages_version 和 conversations 跟随 uid 分片组整体迁移
- **群消息扇出**：扇出目标分散在多个 uid 分片，分片数增加时并发度提升，单分片写入量降低
- **热点用户**：收件箱按 uid 分片，写入随用户分布；msg_id 为 UUIDv7（时间有序 + 随机），索引局部性好。高频群聊可能导致部分分片写入集中，可通过增加分片数缓解

## 8. 改进建议

- **消息撤回**：当前通过 `send_message` + `MESSAGE_TYPE_RECALL` + `RecallBody` 实现：服务端覆盖原消息为撤回占位 body、追加一条 recall 事件消息，并向异步任务队列投递一个 `recalled` 群消息任务（靠 Insert/Update 幂等收敛群成员）；后续可扩展管理员代撤回、审计留痕和更精确的未读补偿。
- **已读回执**：当前实现了会话级未读计数（`unread_count`）和 `clear_unread` 接口。若需精确的消息级已读状态（如"对方已读"提示），可在 conversation 表增加 `read_seq` 字段，客户端上报已读位置
- **扇出优化**：当前群消息扇出在单个 goroutine 组内串行写入。可引入写入队列 + 批量 INSERT 提升吞吐
