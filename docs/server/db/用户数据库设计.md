## 1. 设计目标与约束

- 支持用户注册、登录、资料查看与修改
- 用户名全局唯一，注册后不可修改
- uid 由 Snowflake 生成，全局唯一
- 用户名查找和用户数据分属不同分片组，注册时需跨分片写入

## 2. 核心思想

**双表分离**：将"用户名→UID 映射"和"用户数据"拆分为两张独立表，分属不同分片组。

- `user_lookup`：按 username 分片，解决"已知用户名，查 UID"的路由问题
- `user_info`：按 uid 分片，存储用户的全部属性

这样设计是因为：注册/登录时只知道用户名，需要先查 user_lookup 得到 uid，再路由到 uid 分片读取完整用户信息。

## 3. 表设计

### user_lookup — 用户名查找（username 分片）

```sql
CREATE TABLE IF NOT EXISTS user_lookup (
    username TEXT    PRIMARY KEY,
    uid      INTEGER NOT NULL
);
```

| 字段 | 类型 | 说明 |
|------|------|------|
| username | TEXT PK | 用户名（全局唯一） |
| uid | INTEGER | 对应的用户 ID |

### user_info — 用户资料（uid 分片）

```sql
CREATE TABLE IF NOT EXISTS user_info (
    uid           INTEGER PRIMARY KEY,
    username      TEXT    NOT NULL,
    password_hash TEXT    NOT NULL,
    nickname      TEXT    NOT NULL DEFAULT '',
    avatar        TEXT    NOT NULL DEFAULT '',
    created_at    INTEGER NOT NULL,
    updated_at    INTEGER NOT NULL
);
```

| 字段 | 类型 | 说明 |
|------|------|------|
| uid | INTEGER PK | 用户 ID（Snowflake） |
| username | TEXT | 冗余存储，主查询走 user_lookup |
| password_hash | TEXT | bcrypt 哈希 |
| nickname | TEXT | 昵称 |
| avatar | TEXT | 头像 URL |
| created_at | INTEGER | 注册时间（ms） |
| updated_at | INTEGER | 最后更新时间（ms） |

## 4. 对外接口逻辑

### register — 注册

```
1. UserLookupStore.Insert(username, uid)  — username 分片
   INSERT OR IGNORE，返回 false 说明用户名已占用
2. UserStore.Create(uid, username, hash, nickname, now)  — uid 分片
   INSERT INTO user_info
```

两步写入无事务保护。若步骤 2 失败，user_lookup 中残留孤立记录，由 User GC 定期清理。

### login — 登录

```
1. UserLookupStore.GetUID(username)  — username 分片，O(1)
2. UserStore.Get(uid)                — uid 分片，O(1)，含 password_hash
3. bcrypt.Compare(password, hash)    — CPU 密集
```

### get_user_infos — 批量查询用户资料

```
UserStore.ListByUIDs(uids) — uid 分片，单分片一次 IN 查询，O(N)
返回指定用户 ID 列表的公开资料（不含 password_hash），不存在的 UID 被跳过
```

WebSocket 入口会先对 `uids` 去重；去重后数量超过 `[client].batch_max_limit` 时直接返回 `ok=false`、`error_code=BATCH_LIMIT_EXCEEDED`，调用方应分批重试。

若调用方通讯录中存在这些好友联系人，服务端会在昵称变化时刷新 contact 表的投影：无备注时重算 `sort_key`，并始终重算包含昵称的 `search_text`，相应递增联系人 `seq`。

### update_user_info — 修改资料

```
UserStore.UpdateProfile(uid, nickname, avatar, now) — uid 分片，O(1)
```

服务端 action 名称为 `update_user_info`；当前 DAL 方法仍沿用 `UpdateProfile`，仅表示更新 user_info 表中的资料字段。

### update_password — 修改密码

```
1. UserStore.Get(uid)                       — 验证旧密码
2. UserStore.UpdatePassword(uid, newHash)    — 更新密码
3. 遍历 UserSessionStore.ListTokens(uid)     — 获取所有 token
4. 逐个 SessionStore.Delete(token)           — 删除所有 session
5. UserSessionStore.RemoveTokens(uid)        — 清空 token 索引
```

修改密码后踢下线所有设备。

### search_user — 搜索用户

```
1. UserLookupStore.GetUID(username)  — username 分片
2. UserStore.GetInfo(uid)         — uid 分片
```

精确匹配用户名，跨两个分片组。

## 5. 垃圾回收

**User GC**：清理 user_lookup 中的孤立记录。

场景：注册时步骤 1 成功但步骤 2 失败，user_lookup 中有 username→uid 映射但 user 表无对应记录。

```
间隔：3600s
流程：
1. UserLookupStore.ListAll(limit=500) — 分批扫描所有映射
2. 对每个 (username, uid)：UserStore.GetInfo(uid)
3. 若 user 不存在 → UserLookupStore.Delete(username)
```

## 6. 并发控制

- **user_lookup.Insert**：使用 `INSERT OR IGNORE`，并发注册同一用户名时只有一个成功
- **user 表**：按 uid 分片，不同用户落在不同分片，无竞争
- **同一用户并发修改**：SQLite WAL 模式下写入串行化，自然互斥
- **读写分离**：GetInfo 走 Reader，写操作走 Writer，读不阻塞写

## 7. 扩容策略

- **user_lookup**：按 username 哈希分片，扩容时需迁移映射数据。由于数据量小（每条仅 username+uid），迁移成本低
- **user**：按 uid 哈希分片，扩容需迁移用户记录。Snowflake ID 无序分布，负载天然均衡
- 扩容期间新注册的用户按新分片数路由，老数据通过后台任务逐步迁移

## 8. 改进建议

- **注册原子性**：当前跨分片注册无事务保护，依赖 GC 清理残留。可考虑引入两阶段提交或补偿机制
- **用户名变更**：当前 username 不可变。若需支持改名，需同时更新 user_lookup（删旧建新）和 user.username 字段
- **批量查询**：`ListByUIDs(uids []int64)` 已实现，用于前端 uidCache 填充和 `get_user_infos` 接口
