## 1. 设计目标与约束

- 支持好友关系管理（请求/接受/拒绝/删除）、群联系人收藏和组织成员资格行（好友 / 群 / 组织三类条目同表同机制）
- 支持基于版本号的增量同步（客户端维护 last_sync_seq）
- 支持软删除 + GC 物理清理，保证增量同步不丢失删除事件
- 支持按服务端维护的 `sort_key` 稳定排序、`search_text` 搜索投影，便于通讯录分页化加载和严格控制前端内存
- 好友关系是双向的：A 加 B 为好友，A 和 B 的通讯录各有一条记录
- 群联系人是单向的：用户收藏群组时仅写入自己的通讯录

## 2. 核心思想

### 流即快照

contacts 表同时充当"当前状态快照"和"变更流"。每次操作递增 seq，客户端通过 `WHERE seq > last_sync_seq` 拉取所有变更（包括删除标记）。

无需额外的 changelog 表，一张表同时满足查询和同步两种需求。

### GC 水位线

软删除的记录不能永久保留（浪费存储），但物理删除后客户端可能遗漏删除事件。解决方案：

- `contacts_version.gc_safe_seq`：记录已物理删除的最大 seq
- 客户端同步时若 `last_sync_seq < gc_safe_seq`，说明有数据被清理，需要全量重建
- GC 时先快照待删除记录的最大 seq，再物理删除，最后升 gc_safe_seq

### 排序与搜索投影

contacts 是“通讯录排序/搜索投影 + 同步流”，不是普通 cache。两个投影列：

- `sort_key`：分页排序键。有备注按 `remark_name`、否则按好友 `nickname` 或群 `name` 归一化（首版小写，后续可扩展拼音/首字母）。
- `search_text`：搜索投影。用户联系人拼接 `remark_name` 与 `nickname`，群收藏拼接 `remark_name` 与群 `name`；**明确不含 username**。

不落库 `display_name`：真实展示名由代码按 `remark_name` 与 profile/group info 实时计算。

刷新规则：
- 改备注：重算 `sort_key`/`search_text`，bump `seq`，更新 `updated_at`。
- 昵称/群名变化（`get_user_infos` / `get_group_infos` 时）：无备注则重算 `sort_key`；因为 `search_text` 含昵称/群名，无论是否有备注都重算 `search_text` 并 bump `seq`。
- `username` 不参与投影，用户名变化不触发更新。

## 3. 表设计

### contacts — 联系人（uid 分片）

```sql
CREATE TABLE IF NOT EXISTS contacts (
    uid            INTEGER NOT NULL,
    type           INTEGER NOT NULL,
    id             INTEGER NOT NULL,
    status         INTEGER NOT NULL CHECK (status <> 0),
    remark_name    TEXT    NOT NULL DEFAULT '',
    sort_key       TEXT    NOT NULL DEFAULT '',
    search_text    TEXT    NOT NULL DEFAULT '',
    seq            INTEGER NOT NULL DEFAULT 0,
    created_at     INTEGER NOT NULL DEFAULT 0,
    updated_at     INTEGER NOT NULL DEFAULT 0,
    PRIMARY KEY (uid, type, id)
);
CREATE INDEX IF NOT EXISTS idx_contacts_seq ON contacts(uid, seq);
CREATE INDEX IF NOT EXISTS idx_contacts_sort ON contacts(uid, status, sort_key, type, id);
CREATE INDEX IF NOT EXISTS idx_contacts_search ON contacts(uid, status, search_text);
```

| 字段 | 类型 | 说明 |
|------|------|------|
| uid | INTEGER | 所属用户 |
| type | INTEGER | 目标类型：1=好友，2=群收藏，3=组织 |
| id | INTEGER | 目标 ID：随 `type` 分别表示 `friend_uid`、`group_id` 或 `org_id` |
| status | INTEGER | 1=好友(FRIEND), 2=待确认(PENDING), 0xff=已删除(DELETED) |
| remark_name | TEXT | 备注名 |
| sort_key | TEXT | 排序投影：有备注按备注、否则按昵称/群名/组织名归一化 |
| search_text | TEXT | 搜索投影：拼接 remark_name 与昵称/群名/组织名，不含 username |
| seq | INTEGER | 版本序列号（每次变更递增） |
| created_at / updated_at | INTEGER | 创建/更新时间，毫秒时间戳 |
`type` 与 `id` 组成统一目标键：好友、群收藏、组织成员资格各占一类行；对外协议仍投影为 `friend_uid` / `group_id` / `org_id`。
组织行由组织建制写路径联动维护（入职 Upsert `status=FRIEND`、离职软删除），无 PENDING 态、支持备注，
投影基于组织名（`get_org_infos` 时惰性刷新）；tag 图本身是独立同步域，见 [`../组织架构方案.md`](../组织架构方案.md)。

### contacts_version — 版本控制（uid 分片）

```sql
CREATE TABLE IF NOT EXISTS contacts_version (
    uid         INTEGER PRIMARY KEY,
    gc_safe_seq INTEGER NOT NULL DEFAULT 0,
    max_seq     INTEGER NOT NULL DEFAULT 0
);
```

| 字段 | 类型 | 说明 |
|------|------|------|
| uid | INTEGER PK | 用户 ID |
| gc_safe_seq | INTEGER | GC 安全水位线 |
| max_seq | INTEGER | 当前最大序列号 |

## 4. 对外接口逻辑

### add_friend — 发起好友请求

```
双向写入（各自的 uid 分片）：
1. ContactStore.Upsert(myUID, friendUID, 0, PENDING, remarkName, sortKey, searchText, now) — 我的分片
2. ContactStore.Upsert(friendUID, myUID, 0, PENDING, "", sortKey, searchText, now) — 对方分片
3. 推送 contacts:updated 通知给对方
```

投影由服务端基于 remark_name 与昵称/群名预先算好：`sort_key=ContactSortKey(remark, name)`、`search_text=ContactSearchText(remark, name)`。被申请方不自动写备注，投影基于申请方昵称。

Upsert 内部事务：`bumpSeq(uid)` 递增 max_seq → INSERT OR UPDATE contacts，赋新 seq。

### accept_friend — 接受好友请求

```
双向写入：
1. ContactStore.AcceptRequest(myUID, friendUID)    — PENDING→FRIEND，bump seq
2. ContactStore.AcceptRequest(friendUID, myUID)    — PENDING→FRIEND，bump seq
3. 推送 contacts:updated 通知给对方
```

### reject_friend — 拒绝好友请求

```
双向写入：
1. ContactStore.RejectRequest(myUID, friendUID)    — PENDING→DELETED，bump seq
2. ContactStore.RejectRequest(friendUID, myUID)    — PENDING→DELETED，bump seq
3. 推送 contacts:updated 通知给对方
```

### delete_friend — 删除好友

```
单向操作：
1. ContactStore.Delete(myUID, friendUID, 0)  — 我方好友关系软删除，status→DELETED，bump seq
2. 推送 contacts:updated 通知给自身（用于多端同步）
```

删除好友为单向操作，仅调用方通讯录更新为 DELETED，对方通讯录记录保持不变。

### update_remark — 修改备注名

```
1. ContactStore.UpdateRemark(uid, friendUID, groupID, remarkName, sortKey, searchText, now) — bump seq
```

服务端始终读取展示名（昵称/群名）以重算 `search_text`（含昵称/群名）；`sort_key` 有备注按备注、否则按展示名。

### favorite_group — 收藏群到通讯录

```
1. 校验调用方是群成员
2. ContactStore.Upsert(uid, 0, groupID, FRIEND, remarkName, sortKey, searchText, now) — bump seq
```

群收藏没有备注时 `sort_key`/`search_text` 基于群名。

### unfavorite_group — 取消群收藏

```
单向操作：
1. ContactStore.Delete(uid, 0, groupID) — 群收藏软删除，status→DELETED，bump seq
2. 推送 contacts:updated 通知给自身（用于多端同步）
```

取消群收藏只影响调用方自己的通讯录。

### get_contacts — 列出联系人

```
ContactStore.ListFiltered(uid, filter, offset, limit)
→ SELECT ... WHERE uid = ? AND status != DELETED
    AND status/friend_uid/group_id filters...
    ORDER BY
      CASE status=PENDING: seq DESC, friend_uid ASC
      ELSE: sort_key COLLATE NOCASE ASC, friend_uid ASC, group_id ASC
    LIMIT ? OFFSET ?
```

接口支持 `status?`、`friend_uid?`、`group_id?` 过滤，并返回：

- `contacts[]`：当前分页数据，单条含 `friend_uid/group_id/status/seq/remark_name/sort_key/search_text`
- `total`：过滤后的总数（`PageInfo.total`，可为 -1 表示未统计）；前端列表按"已加载窗口"渲染，不依赖 total 撑高滚动区
- `seq`：当前 contacts_version.max_seq，供客户端初始化或推进联系人增量游标

### sync_contacts — 增量同步

客户端传入 last_sync_seq：
- 若 `rebuild=false && last_sync_seq < gc_safe_seq`：返回 `seq_too_old` 错误；持久存储客户端清空本地联系人表和游标后，从 `last_sync_seq=0, rebuild=true` 重新执行 `sync_contacts`
- 增量拉取 `WHERE seq > last_sync_seq`，仅返回 `contacts[]`；`status=0xff` 删除 tombstone 用于推进游标，持久存储客户端收到后删除对应本地行，不保存 tombstone 行
- `sort_key`/`search_text` 变更也会 bump `seq`，因此无备注联系人昵称/群名变化、或有备注联系人昵称/群名变化（影响 search_text）后都能通过增量同步收敛

联系人同步与其他同步域共同遵守 [`../../同步机制方案.md`](../../同步机制方案.md) 的轻通知、分页读取、游标增量和过旧游标重建规则；本文只维护 contacts 表、`contacts_version` 和排序/搜索投影细节。

## 5. 垃圾回收

**Contact GC**：物理删除软删除记录，升 GC 水位线。

```
默认间隔：3600s（若运行时配置值 <= 0，内部兜底为 300s）
流程：
1. ContactStore.ListPurgeable(limit=500, afterUID)
  → SELECT DISTINCT uid FROM contacts WHERE status = 0xff AND uid > ? LIMIT ?
2. 对每个 uid：
   ContactStore.Purge(uid) — 事务内三步：
  a. 快照：SELECT MAX(seq) FROM contacts WHERE uid = ? AND status = 0xff
  b. 物理删除：DELETE FROM contacts WHERE uid = ? AND status = 0xff
  c. 升水位线：UPDATE contacts_version SET gc_safe_seq = MAX(old, snapshotSeq) WHERE uid = ?
```

事务保证三步原子性。gc_safe_seq 单调递增，不会回退。

## 6. 并发控制

- **seq 分配**：`bumpSeq` 在事务内执行 `UPDATE contacts_version SET max_seq = max_seq + 1 RETURNING max_seq`，SQLite 单写保证原子
- **双向写入无事务**：A 和 B 的通讯录可能在不同分片，无跨分片事务。若一侧写入失败，另一侧可能出现不对称状态（A 有 B 但 B 没 A），业务层容忍
- **GC 与正常操作**：Purge 在事务内完成，与 Upsert/Delete 等操作串行化（同一分片的 Writer）
- **GC 与同步并发**：同步走 Reader，GC 走 Writer。WAL 模式下读不阻塞写。极端情况下同步可能读到即将被删除的记录，但 gc_safe_seq 升高后客户端下次同步会全量重建

## 7. 扩容策略

- **contacts + contacts_version**：跟随 uid 分片组整体扩容
- **双向写入**：A 加 B 为好友需写两个 uid 分片，扩容后两侧可能在不同节点，写入延迟略增但不影响正确性
- **GC 扫描**：ListPurgeable 使用游标分页，分片数增加时每个分片扫描量减少

## 8. 改进建议

- **好友数量限制**：当前无上限。可在 Upsert 时检查该 uid 的好友数量（`COUNT WHERE status = FRIEND`），超限拒绝
- **双向一致性**：当前双向写入无事务保护。可引入异步补偿任务：定期扫描不对称记录并修复
