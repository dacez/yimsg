## 1. 设计目标与约束

- 基于 token 的服务端 session 管理，非 JWT
- 支持多设备同时登录
- 支持单设备登出、全设备踢下线（修改密码时）
- token 随机生成（128-bit），天然均匀分布
- session 有过期时间，需定期清理

## 2. 核心思想

**双表互为索引**：session 按 token 分片（快速鉴权），user_session 按 uid 分片（快速查找用户所有 token）。

- `session`：已知 token，查 uid + 过期时间 → 鉴权
- `user_session`：已知 uid，查所有 token → 踢下线

两张表的数据需保持同步：创建时双写，删除时双删。若不一致，由 GC 清理孤立记录。

## 3. 表设计

### session — 认证会话（token 分片）

```sql
CREATE TABLE IF NOT EXISTS session (
    token      TEXT    PRIMARY KEY,
    uid        INTEGER NOT NULL,
    created_at INTEGER NOT NULL,
    expire_at  INTEGER NOT NULL
);
CREATE INDEX IF NOT EXISTS idx_session_uid ON session(uid);
```

| 字段 | 类型 | 说明 |
|------|------|------|
| token | TEXT PK | 随机生成的认证令牌 |
| uid | INTEGER | 所属用户 ID |
| created_at | INTEGER | 创建时间（ms） |
| expire_at | INTEGER | 过期时间（ms） |

`idx_session_uid`：按 uid 查找用户所有 session，用于 `DeleteByUID` 踢下线。

### user_session — 用户 token 索引（uid 分片）

```sql
CREATE TABLE IF NOT EXISTS user_session (
    uid        INTEGER NOT NULL,
    token      TEXT    NOT NULL,
    device     TEXT    NOT NULL DEFAULT '',
    created_at INTEGER NOT NULL,
    PRIMARY KEY (uid, token)
);
```

| 字段 | 类型 | 说明 |
|------|------|------|
| uid | INTEGER | 用户 ID |
| token | TEXT | 认证令牌 |
| device | TEXT | 设备标识 |
| created_at | INTEGER | 创建时间（ms） |

## 4. 对外接口逻辑

### authenticate — 鉴权

```
1. SessionStore.Get(token)  — token 分片，O(1)
2. 检查 expire_at > now
3. 返回 uid
```

每次 WebSocket 请求都走这条路径（连接建立时调用一次）。

### login — 创建 session

```
1. 生成随机 token
2. UserSessionStore.AddToken(uid, token, device, now)  — uid 分片
3. SessionStore.Create(token, uid, now, expireAt)       — token 分片
```

双写无事务保护。若步骤 3 失败，user_session 中残留孤立记录，由 GC 清理。

### logout — 登出

```
1. SessionStore.Get(token)                        — token 分片，获取 uid
2. UserSessionStore.RemoveToken(uid, token)        — uid 分片
3. SessionStore.Delete(token)                      — token 分片
```

### update_password — 全设备踢下线

```
1. UserSessionStore.ListTokens(uid)     — uid 分片，获取所有 token
2. 遍历每个 token：
   SessionStore.Delete(token)          — token 分片（可能跨多个分片）
3. UserSessionStore.RemoveTokens(uid)  — uid 分片，一次性清空
```

## 5. 垃圾回收

**Session GC**：两阶段清理。

### 阶段 1：清理过期 session

```
默认间隔：3600s（若运行时配置值 <= 0，内部兜底为 60s）
流程：遍历所有 token 分片
  SessionStore.Purge(now, limit=500)
  → DELETE FROM session WHERE token IN (SELECT token FROM session WHERE expire_at <= ? LIMIT ?)
```

### 阶段 2：清理孤立 user_session

```
流程：遍历所有 uid 分片
  UserSessionStore.ListAll(limit=500) → 获取有 token 的 uid 列表
  对每个 uid：
    UserSessionStore.ListTokens(uid) → 获取所有 token
    对每个 token：
      SessionStore.Get(token) → 检查 session 是否存在
      若不存在 → UserSessionStore.RemoveToken(uid, token)
```

孤立 user_session 产生场景：
- Purge 删除了过期 session，但 user_session 未同步删除
- 登出时步骤 2 成功但步骤 3 失败（极少见）

## 6. 并发控制

- **token 全局唯一**：随机 128-bit，碰撞概率可忽略
- **同一用户多设备登录**：各自创建独立 session，互不影响
- **踢下线与鉴权并发**：鉴权读 session 表（Reader），踢下线写 session 表（Writer）。WAL 模式下读不阻塞写，最坏情况是鉴权读到即将被删除的 session（短暂延迟，可接受）
- **GC 与正常操作并发**：Purge 仅删除 expire_at <= now 的记录，正常操作的 session 不受影响

## 7. 扩容策略

- **token 分片**：token 随机生成，天然均匀分布，扩容效果好
- **uid 分片**（user_session）：跟随 uid 分片组整体扩容
- **跨分片操作**：踢下线时需遍历多个 token 分片删除 session，分片数增加时删除次数不变（每个 token 只在一个分片），但并发度可提升

## 8. 改进建议

- **session 续期**：当前有 `Renew` 接口但未在业务层调用。可在每次鉴权成功时自动续期，减少用户频繁重新登录
- **设备数量限制**：当前无最大设备数限制。可在 AddToken 时检查该 uid 的 token 数量，超限时踢掉最早的设备
- **批量创建 session 优化**：login 的双写可改为先写 session（关键路径），user_session 异步补写，减少登录延迟
