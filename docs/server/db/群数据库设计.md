## 1. 设计目标与约束

- 支持群组创建、信息修改、成员增删
- 群属性和成员数据路由到同一分片（group_id 分片），保证群操作的局部性
- 成员增删是并发安全的（无需读-改-写）
- 群消息的发送和存储在消息模块完成（uid 分片），群模块只负责群元数据和成员管理

## 2. 核心思想

### 双表拆分

将群信息和群成员拆分为两张独立表：

- `group_info`：群名称、头像、群主等属性
- `group_member`：每个成员一行，独立的 INSERT/DELETE

若用单表 + JSON 数组存成员列表，并发添加/删除成员需要读-改-写，存在竞态条件。双表设计下：
- 加人 = `INSERT INTO group_member`
- 踢人 = `DELETE FROM group_member`
- 改群名 = `UPDATE group_info`

三种操作互不干扰，无需加锁。

### 群消息分离

群消息不存储在 group 分片，而是扇出写入每个成员的 uid 分片（见消息数据库设计）。群模块只在发送消息时提供成员列表（`ListAllMembers`），消息存储完全在 uid 分片完成。

## 3. 表设计

### group_info — 群信息（group 分片）

```sql
CREATE TABLE IF NOT EXISTS group_info (
    group_id   INTEGER PRIMARY KEY,
    name       TEXT    NOT NULL DEFAULT '',
    avatar     TEXT    NOT NULL DEFAULT '',
    owner_uid  INTEGER NOT NULL,
    created_at INTEGER NOT NULL,
    updated_at INTEGER NOT NULL
);
```

| 字段 | 类型 | 说明 |
|------|------|------|
| group_id | INTEGER PK | 群 ID（Snowflake） |
| name | TEXT | 群名称 |
| avatar | TEXT | 群头像 URL |
| owner_uid | INTEGER | 群主 UID |
| created_at | INTEGER | 创建时间（ms） |
| updated_at | INTEGER | 最后更新时间（ms） |

### group_member — 群成员（group 分片）

```sql
CREATE TABLE IF NOT EXISTS group_member (
    group_id  INTEGER NOT NULL,
    uid       INTEGER NOT NULL,
    role      INTEGER NOT NULL DEFAULT 0,
    joined_at INTEGER NOT NULL,
    PRIMARY KEY (group_id, uid)
);
```

| 字段 | 类型 | 说明 |
|------|------|------|
| group_id | INTEGER | 群 ID |
| uid | INTEGER | 成员 UID |
| role | INTEGER | 0=普通成员(MEMBER), 2=群主(OWNER) |
| joined_at | INTEGER | 加入时间（ms） |

## 4. 对外接口逻辑

### create_group — 创建群组

```
事务内执行（group 分片）：
1. INSERT INTO group_info (groupID, name, ownerUID, now)
2. 遍历 memberUIDs：INSERT INTO group_member (groupID, uid, role, now)
   — 群主 role=OWNER(2)，其他 role=MEMBER(0)
3. 若 ownerUID 不在 memberUIDs 中，自动加入

事务后：
4. 发送系统消息给所有成员（扇出写入各自 uid 分片）
```

### get_group_info — 查看群信息

```
GroupStore.GetInfo(groupID) — group 分片，O(1)
```

### get_group_infos — 批量查询群信息

```
GroupStore.ListByIDs(groupIDs) — group 分片，单分片一次 IN 查询，O(N)
返回指定群 ID 列表的群信息，不存在的 ID 被跳过
```

WebSocket 入口会先对 `group_ids` 去重；去重后数量超过 `[client].batch_max_limit` 时直接返回 `ok=false`、`error_code=BATCH_LIMIT_EXCEEDED`，调用方应分批重试。

若调用方通讯录中存在这些群收藏，服务端会在群名变化时刷新 contact 表的投影：无备注时重算 `sort_key`，并始终重算包含群名的 `search_text`，相应递增联系人 `seq`。

### get_group_members — 分页获取群成员

```
GroupStore.ListMembers(groupID, offset, limit) + GroupStore.CountMembers(groupID) — group 分片，O(limit)
按 joined_at 排序，分页返回；`limit` 单次最多 500
```

仅返回 `uid, role, joined_at`，显示数据（昵称/头像）由客户端 uidCache 解决。

### update_group_info — 修改群信息

```
GroupStore.UpdateInfo(groupID, name, avatar, now) — group 分片，O(1)
```

### add_group_member — 拉人进群

```
1. GroupStore.AddMember(groupID, uid, MEMBER, now) — INSERT OR IGNORE
   返回 false 说明已是成员
2. GroupStore.ListAllMembers(groupID) — 获取当前所有成员
3. 发送系统消息 "XXX joined" 给所有成员
```

### remove_group_member — 踢人出群

```
1. GroupStore.ListAllMembers(groupID) — 获取移除前的成员列表（用于通知）
2. GroupStore.RemoveMember(groupID, targetUID) — DELETE
   返回 false 说明不是成员
3. 发送系统消息 "XXX removed" 给所有成员（含被踢者）
```

## 5. 垃圾回收

群数据无 GC 需求：

- **group_info**：群创建后持久存在，无过期概念
- **group_member**：成员退出即物理删除（DELETE），无软删除

若需支持群解散，可在 group_info 增加 `status` 字段标记，但当前未实现。

## 6. 并发控制

- **创建群组**：事务内完成 group_info + group_member 写入，原子性有保证
- **并发加人**：`INSERT OR IGNORE` 保证幂等，两个请求同时加同一个人只有一个成功
- **并发踢人**：DELETE 走主键，幂等操作
- **成员变更与消息发送**：发群消息时先调 `ListAllMembers` 获取成员快照，即使在扇出过程中有人被踢，消息已写入的部分不会回滚（最终一致）
- **群信息修改与成员操作**：group_info 和 group_member 是独立表，修改群名和加/踢人互不阻塞（同一分片的 Writer 会串行化，但操作极快）

## 7. 扩容策略

- **group 分片扩容**：group_info 和 group_member 按 group_id 路由，同一个群的所有数据始终在同一分片
- **大群问题**：一个万人群的 group_member 有 1 万行，全在一个分片。`ListAllMembers` 全量查询需 O(M)。对外接口已改为分页（`ListMembers`），内部扇出仍用全量查询
- **跨分片操作**：群消息扇出需写入 M 个 uid 分片，分片数增加时并发度提升

## 8. 改进建议

- **群解散**：当前无解散功能。可在 group_info 增加 `status` 字段，解散时标记而非删除
- **成员角色扩展**：当前仅有 MEMBER(0) 和 OWNER(2)。可扩展 ADMIN(1) 角色，支持管理员权限
- **成员数量限制**：当前无上限。可在 AddMember 时 COUNT 检查，超限拒绝
- **群主转让**：当前无转让接口。可新增 `TransferOwner` 操作，事务内同时修改 group_info.owner_uid 和两个 group_member.role
- **ListAllMembers 性能**：内部扇出用的 `ListAllMembers` 全量查询大群时可能成为瓶颈。可考虑缓存成员列表或引入成员变更通知机制
