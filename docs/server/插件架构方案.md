## 概览

Yimsg 保留插件架构，用于把可选能力与核心 IM 能力分离。插件可以贡献 schema、监听断连事件、启动后台任务，并通过统一 Host 接口访问核心服务能力。插件不提供按请求分发（per-request dispatch）的扩展点——任何需要对外暴露的请求/响应能力，都必须在 `internal/protocol/yimsg.proto` 里定义真正的 protobuf action，同 `protocolgen` 生成物走同一条编解码路径，详见 [协议扩展机制](#协议扩展机制)。

核心 IM 功能仍保留在主服务中：认证、联系人、屏蔽列表、消息、群组、会话、上传和推送基础设施。插件系统只提供扩展点，不把某个可选能力固化进核心。

## 后端插件架构

### Plugin 接口

位于 `internal/plugin/plugin.go`，定义插件必须实现的接口：

```go
type Plugin interface {
    // Name 返回插件唯一标识符
    Name() string

    // Schemas 返回插件需要的额外 DDL，按 shard group 分组
    // key: "uid" | "username" | "token" | "group"
    // value: DDL 字符串（CREATE TABLE ... 等）
    Schemas() map[string]string

    // OnDisconnect 在用户 WebSocket 断连时被调用（可选钩子）
    OnDisconnect(host Host, uid int64)

    // OnStart 在服务器启动完成后调用，用于启动后台任务（GC、定时器等，可选）
    OnStart(host Host)
}
```

插件接口只提供 schema 贡献和两个生命周期钩子，不提供按请求分发的扩展点。插件如果需要处理 WebSocket 请求，必须像核心业务一样在 `internal/protocol/yimsg.proto` 里定义 action 与 Request/Response message，由 `protocolgen` 生成的分发代码调用到实现方法上——与核心 action 走完全相同的路径，没有插件专属的旁路。

### Host 接口

为避免循环依赖（`plugin` ← `service`），插件通过 `Host` 接口访问服务端能力：

```go
type Host interface {
    DB() *shard.Database
    IDGen() *snowflake.Generator
    Config() *config.Config
    Online() *online.Registry

    UserStore(uid int64) dal.UserStoreAPI
    ContactStore(uid int64) dal.ContactStoreAPI
    BlocklistStore(uid int64) dal.BlocklistStoreAPI
    MessageStore(uid int64) dal.MessageStoreAPI
    ConversationStore(uid int64) dal.ConversationStoreAPI
    MutelistStore(uid int64) dal.MutelistStoreAPI
    UserSessionStore(uid int64) dal.UserSessionStoreAPI
    GroupStore(groupID int64) dal.GroupStoreAPI
    SessionStore(token string) dal.SessionStoreAPI
    UserLookupStore(username string) dal.UserLookupStoreAPI

    IsEitherWayBlocked(a, b int64) (bool, error)
}
```

`service.AppState` 实现 `Host` 接口，作为插件宿主。

### Registry

位于 `internal/plugin/registry.go`，负责插件注册、schema 合并与两个生命周期钩子的广播：

```go
type Registry struct {
    plugins map[string]Plugin
}

func NewRegistry() *Registry
func (r *Registry) Register(p Plugin)
func (r *Registry) MergeSchemas(base map[string]string) map[string]string
func (r *Registry) HandleDisconnect(host Host, uid int64)
func (r *Registry) Start(host Host)
```

注册时只检查插件名冲突（重名 panic）；schema 合并时会把核心 DDL 与插件 DDL 按分片组拼接后交给 `shard.Open()` 初始化。`Registry` 不维护 action 到插件的映射，也没有 `Dispatch` 方法——WebSocket 分发完全由 `internal/ws/action_dispatch_gen.go`（`protocolgen` 生成）按 action 直接路由到 `AppState` 上的方法，插件没有独立的分发入口。

## 协议扩展机制

线上协议是纯二进制 protobuf 帧（见 [`../protocol/接口总览.md`](../protocol/接口总览.md)），单一事实源是 `internal/protocol/yimsg.proto`。这条通道不区分"核心"和"插件"：任何 action / notification，不管由核心业务还是插件贡献，都必须先在 proto 里定义独立的 Request / Response（或 notification message），跑 `go run ./tools/cmd/protocolgen` 刷新生成物，再由生成的分发代码路由到实现方法。

插件系统本身**不提供**任何绕过这条路径的旁路——没有"插件专属的 JSON 字段展平"、没有 `Request.Raw`/`Response.Data` 这类任意载荷透传机制、没有独立于 `protocolgen` 的插件请求分发表。`appmsg.Request`/`appmsg.Response` 是早期测试辅助遗留的字面量类型，只服务于部分 `internal/service/*_test.go` 通过 `responseFromProto` 做字段级断言，不是任何真实的请求/响应载体，插件代码不应该依赖它们。

插件如需暴露新的 WebSocket 能力，落地步骤与新增一个核心 action 完全一致：改 proto → 跑 protocolgen → 在 `service.AppState` 上实现对应方法 → 补测试与文档。`internal/ws/connection.go` 当前把 `*service.AppState` 直接作为具体分发目标（`DispatchActionFrame(state, ...)`），不存在按插件切换分发目标的机制，因此插件新增的 action 目前需要落在 `internal/service/` 包内，与核心业务代码同等对待，不是独立注册的旁路能力。复杂数据通过独立的 Request/Response 字段传递，notification 只带轻量提示字段（如 `org:updated` 只带 `org_id`），不携带任意结构的增量数据。

## 主服务端集成

`cmd/server/main.go` 始终创建插件注册中心，并把它传入 `service.AppState`：

```go
registry := plugin.NewRegistry()

baseSchemas := dal.Schemas()
allSchemas := registry.MergeSchemas(baseSchemas)

db, err := shard.Open(cfg.Database.DataDir, cfg.Database.ShardCount, allSchemas)
state := service.NewAppState(db, cfg, registry)

registry.Start(state)
```

当前没有默认注册的业务插件。后续新增插件时，只需要在打开数据库前完成 `registry.Register(plugin)`，使其 schema 能参与初始化；`registry.Start(state)` 会在服务启动完成后调用每个插件的 `OnStart` 钩子。

`internal/ws/connection.go` 的 WebSocket 分发（`DispatchActionFrame`，`internal/ws/action_dispatch_gen.go` 生成）按 protobuf `type` 字段直接路由到 `*service.AppState` 上的方法，未命中的 `type` 直接返回"unknown action"错误响应，不存在插件兜底分发路径——`Registry` 没有 `Dispatch` 方法（见 [Registry](#registry)）。

### 断连与后台任务

WebSocket 断连时，连接层会调用：

```go
state.Plugins.HandleDisconnect(state, conn.uid)
```

服务启动完成后，主入口会调用：

```go
registry.Start(state)
```

因此插件可以把在线态清理和后台循环放在统一扩展点里。

## 插件开发指南

### 后端新插件模板

插件本身（`Plugin` 接口的实现）只贡献 schema 和生命周期钩子，不携带任何 action 处理逻辑——这部分职责边界与 [概览](#概览) 一致：

```go
package demo

import "yimsg/internal/plugin"

type Plugin struct{}

func New() *Plugin { return &Plugin{} }

func (p *Plugin) Name() string { return "demo" }

func (p *Plugin) Schemas() map[string]string {
    return map[string]string{
        "uid": `CREATE TABLE IF NOT EXISTS demo_note (
            uid INTEGER NOT NULL,
            note_id INTEGER NOT NULL,
            content TEXT NOT NULL DEFAULT '',
            PRIMARY KEY (uid, note_id)
        );`,
    }
}

func (p *Plugin) OnDisconnect(host plugin.Host, uid int64) {}
func (p *Plugin) OnStart(host plugin.Host) {}
```

如果这个插件还需要对外暴露一个 `demo_create_note` action，走的是与核心业务完全相同的路径（见 [协议扩展机制](#协议扩展机制)），不是 `Plugin` 接口的一部分：

1. 在 `internal/protocol/yimsg.proto` 里加 `TYPE_ACTION_DEMO_CREATE_NOTE` 枚举值与 `DemoCreateNoteRequest`/`DemoCreateNoteResponse` message。
2. 跑 `go run ./tools/cmd/protocolgen`，生成物会在 `ActionService` 接口上要求一个新方法。
3. 在 `internal/service/` 包内实现 `func (s *AppState) DemoCreateNote(info *BaseInfo, req *pb.DemoCreateNoteRequest) *pb.DemoCreateNoteResponse`，内部按需调用 `s.IDGen()`、`s.UserStore(uid)` 等既有能力（插件的 `demo_note` 表可以用普通 `database/sql` 或新增一个 DAL Store 访问，与核心表访问方式一致）。

### 前端插件组织

当前前端不保留空插件占位目录。新增 SDK 插件时再创建具体模块，建议保持为纯函数模块：接收 `WsTransport` 和扁平参数，返回 SDK 内部原始结果；是否包装为 `YimsgClient` 公共方法，再按业务 API 稳定性决定。

## 测试策略

- `internal/plugin/plugin_test.go`：插件名冲突检测（`TestRegisterAndNameConflict`）、schema 合并、断连和启动钩子。
- 新增后端插件时，应在插件包内补充单元测试；若插件按 [协议扩展机制](#协议扩展机制) 新增了 action，比照核心 action 的测试规格——`internal/service/*_test.go` 覆盖权限与业务逻辑、`tests/e2e/` 覆盖端到端可用性。
- 新增前端 SDK 插件时，应补充 `frontend/tests/unit/sdk/` 或 `frontend/tests/integration/` 覆盖。
