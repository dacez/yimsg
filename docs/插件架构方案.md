## 概览

Yimsg 保留插件架构，用于把可选能力与核心 IM 能力分离。插件可以贡献 schema、注册 action 处理器、监听断连事件、启动后台任务，并通过统一 Host 接口访问核心服务能力。

核心 IM 功能仍保留在主服务中：认证、联系人、屏蔽列表、消息、群组、会话、上传和推送基础设施。插件系统只提供扩展点，不把某个可选能力固化进核心。

## 后端插件架构

### Plugin 接口

位于 `internal/plugin/plugin.go`，定义插件必须实现的接口：

```go
type Plugin interface {
    // Name 返回插件唯一标识符
    Name() string

    // Schemas 返回插件需要的额外 DDL，按 shard group 分组
    // key: "uid" | "username" | "token" | "group"
    // value: DDL 字符串（CREATE TABLE ... 等）
    Schemas() map[string]string

    // Actions 返回插件注册的 action 名称 → handler 映射
    Actions() map[string]Handler

    // OnDisconnect 在用户 WebSocket 断连时被调用（可选钩子）
    OnDisconnect(host Host, uid int64)

    // OnStart 在服务器启动完成后调用，用于启动后台任务（GC、定时器等，可选）
    OnStart(host Host)
}

// Handler 是插件 action 处理函数签名。
type Handler func(host Host, reqID uint64, uid int64, req *appmsg.Request) *appmsg.Response
```

### Host 接口

为避免循环依赖（`plugin` ← `service`），插件通过 `Host` 接口访问服务端能力：

```go
type Host interface {
    DB() *shard.Database
    IDGen() *snowflake.Generator
    Config() *config.Config
    Online() *online.Registry

    UserStore(uid int64) dal.UserStoreAPI
    ContactStore(uid int64) dal.ContactStoreAPI
    BlocklistStore(uid int64) dal.BlocklistStoreAPI
    MessageStore(uid int64) dal.MessageStoreAPI
    ConversationStore(uid int64) dal.ConversationStoreAPI
    MutelistStore(uid int64) dal.MutelistStoreAPI
    UserSessionStore(uid int64) dal.UserSessionStoreAPI
    GroupStore(groupID int64) dal.GroupStoreAPI
    SessionStore(token string) dal.SessionStoreAPI
    UserLookupStore(username string) dal.UserLookupStoreAPI

    IsEitherWayBlocked(a, b int64) (bool, error)
}
```

`service.AppState` 实现 `Host` 接口，作为插件宿主。

### Registry

位于 `internal/plugin/registry.go`，负责插件注册、schema 合并、action 分发：

```go
type Registry struct {
    plugins map[string]Plugin
    actions map[string]Plugin
}

func NewRegistry() *Registry
func (r *Registry) Register(p Plugin)
func (r *Registry) MergeSchemas(base map[string]string) map[string]string
func (r *Registry) Dispatch(host Host, reqID uint64, uid int64, req *appmsg.Request) (*appmsg.Response, bool)
func (r *Registry) HandleDisconnect(host Host, uid int64)
func (r *Registry) Start(host Host)
```

注册时会检查插件名和 action 名冲突；schema 合并时会把核心 DDL 与插件 DDL 按分片组拼接后交给 `shard.Open()` 初始化。

## 协议扩展机制

### 总原则：核心 WebSocket 通道保持 protobuf

核心 WebSocket action / notification 通道以 `internal/protocol/yimsg.proto` 为单一事实源。核心通知链路不再提供 `Notification.Data` 或任意 JSON 展平扩展；服务端内部传递 `appmsg.Notification` 结构体，出站直接编码 protobuf notification frame。

插件如需扩展对外能力，应优先新增明确的 protobuf action / message，并同步生成物、文档与测试；复杂数据通过 action 拉取，不通过 notification 携带任意 JSON payload。历史 `Response.Data json.RawMessage` 仅保留给已有插件 action 的返回兼容，不作为新增核心接口的设计方式。

### Request

客户端照常发送扁平 JSON。插件需要访问自定义参数时，从 `Request.Raw` 中反序列化自己的结构体：

```go
type Request struct {
    Action    string          `json:"action"`
    RequestID uint64          `json:"request_id"`
    Raw       json.RawMessage `json:"-"`
}
```

### Response

```go
type Response struct {
    RequestID uint64          `json:"request_id"`
    OK        bool            `json:"ok"`
    Error     string          `json:"error,omitempty"`
    Data      json.RawMessage `json:"-"`
}

func OKWithData(reqID uint64, v any) *Response
```

示例：

```go
return appmsg.OKWithData(reqID, map[string]any{
    "demo_id": "demo-1",
    "status":  "ready",
})
```

序列化结果会把 `demo_id`、`status` 与 `request_id`、`ok` 放在同一层。

### Notification

当前核心仓库不再支持插件通过 `Notification.Data` 注入任意 JSON 通知字段。插件通知若需要进入 WebSocket 主链路，必须先在 protobuf 中定义明确 notification message，并由 protocolgen 生成 Go / TS 编解码代码。

## 主服务端集成

`cmd/server/main.go` 始终创建插件注册中心，并把它传入 `service.AppState`：

```go
registry := plugin.NewRegistry()

baseSchemas := dal.Schemas()
allSchemas := registry.MergeSchemas(baseSchemas)

db, err := shard.Open(cfg.Database.DataDir, cfg.Database.ShardCount, allSchemas)
state := service.NewAppState(db, cfg, registry)

registry.Start(state)
```

当前没有默认注册的业务插件。后续新增插件时，只需要在打开数据库前完成 `registry.Register(plugin)`，使其 schema 能参与初始化。

### WebSocket Dispatch

核心 action 仍由 `internal/ws/connection.go` 直接分发。未命中核心 action 时，dispatch 会尝试插件注册中心：

```go
default:
    if resp, handled := s.Plugins.Dispatch(s, reqID, conn.uid, req); handled {
        return dispatchResult{Response: resp}
    }
    return dispatchResult{Response: appmsg.ErrUnknownAction(reqID, "unknown action: "+req.Action)}
```

### 断连与后台任务

WebSocket 断连时，连接层会调用：

```go
state.Plugins.HandleDisconnect(state, conn.uid)
```

服务启动完成后，主入口会调用：

```go
registry.Start(state)
```

因此插件可以把在线态清理和后台循环放在统一扩展点里。

## 插件开发指南

### 后端新插件模板

```go
package demo

import (
    "encoding/json"
    "yimsg/internal/plugin"
    "yimsg/internal/appmsg"
)

type Plugin struct{}

func New() *Plugin { return &Plugin{} }

func (p *Plugin) Name() string { return "demo" }

func (p *Plugin) Schemas() map[string]string {
    return map[string]string{
        "uid": `CREATE TABLE IF NOT EXISTS demo_note (
            uid INTEGER NOT NULL,
            note_id INTEGER NOT NULL,
            content TEXT NOT NULL DEFAULT '',
            PRIMARY KEY (uid, note_id)
        );`,
    }
}

func (p *Plugin) Actions() map[string]plugin.Handler {
    return map[string]plugin.Handler{
        "demo_create_note": p.handleCreateNote,
    }
}

func (p *Plugin) OnDisconnect(host plugin.Host, uid int64) {}
func (p *Plugin) OnStart(host plugin.Host) {}

func (p *Plugin) handleCreateNote(host plugin.Host, reqID uint64, uid int64, req *appmsg.Request) *appmsg.Response {
    var params struct {
        Content string `json:"content"`
    }
    if err := json.Unmarshal(req.Raw, &params); err != nil {
        return appmsg.ErrInvalidArgument(reqID, "invalid payload")
    }
    noteID := host.IDGen().NextID()
    return appmsg.OKWithData(reqID, map[string]any{"note_id": noteID})
}
```

### 前端插件组织

当前前端不保留空插件占位目录。新增 SDK 插件时再创建具体模块，建议保持为纯函数模块：接收 `WsTransport` 和扁平参数，返回 SDK 内部原始结果；是否包装为 `YimsgClient` 公共方法，再按业务 API 稳定性决定。

## 测试策略

- `internal/plugin/plugin_test.go`：注册、冲突检测、分发、schema 合并、断连和启动钩子。
- 新增后端插件时，应在插件包内补充单元测试，并按需增加 `tests/e2e/` 黑盒测试。
- 新增前端 SDK 插件时，应补充 `frontend/tests/unit/sdk/` 或 `frontend/tests/integration/` 覆盖。
