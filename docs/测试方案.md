## 1. 文档范围与口径

研发阶段，以当前分支代码为准。本文说明测试分层、执行入口和验证口径，目标是让测试说明与实现保持一致。

统计口径由 `tools/check_docs_consistency.sh` 统一输出；执行脚本后，在终端输出的“测试用例统计”段查看当前数量，文档不手写具体数量：

- Go 测试按 `func TestXxx` 统计。
- `tests/e2e/setup_test.go` 中 `TestMain` 记为启动入口，不计入业务用例数。
- 前端测试按 `it(...)`/`test(...)` 统计。

## 3. 全量测试执行（推荐）

仓库已提供一键全量脚本，包含：

1. 构建并启动服务端
2. 刷新协议生成物
3. 执行只读文档一致性校验
4. 执行 Go 非 E2E 测试
5. 执行 Go E2E 测试
6. 执行前端 unit / sdk / ui 测试

命令：

```bash
./tools/run_all_tests.sh
```

> 脚本已避免依赖 `mapfile` / `readarray` 等 Bash 4+ 内建，默认兼容 macOS 自带 Bash 3.2、常见 Linux 云主机 Bash 4+，以及 Git Bash / MSYS2 / Cygwin。

只读文档一致性校验也可单独执行：

```bash
./tools/check_docs_consistency.sh
```

协议生成物可单独刷新：

```bash
go run ./tools/cmd/protocolgen
```

协议生成物也可单独做只读校验：

```bash
go run ./tools/cmd/protocolgen --check
```

协议生成器读取 `internal/protocol/yimsg.proto`，通过 `protoc-gen-go` 刷新 `internal/protocol/pb/yimsg.pb.go`，通过 `ts-proto` 刷新 `frontend/src/sdk/generated/yimsg.ts`。全量脚本会先安装固定版本的 `protoc-gen-go` 到 `$(go env GOPATH)/bin`，再在前端构建和服务端启动前执行刷新命令，保证临时环境和内存测试环境也能基于最新 protobuf 生成物运行；`type` 分发和 action 映射由服务端 WS 层、前端 transport 层的 switch 代码维护。

该脚本只读取仓库文件，不修改代码或文档，覆盖 `schema.go` ↔ schema 字段 / 索引对照、WebSocket action ↔ 接口总览、`YimsgClient` 公开方法 ↔ SDK 接口说明，以及 `docs/**/*.md` 的四行头部模板和相对链接。通过时会输出当前表数、字段数、索引数、WebSocket action 数、HTTP 接口数、SDK 公开方法数、文档文件数和测试用例统计，便于复核清单是否漂移。若文档内容有变动但 `最后复核` 日期未更新，脚本只输出提示，不阻断本地迭代。

验证路径选择：

| 变更范围 | 最小校验 | 说明 |
|---|---|---|
| 只改文档、索引、评价或历史内容处理规则 | `./tools/check_docs_consistency.sh` | 只读校验即可确认链接入口、头部模板和自动清单 |
| 改 action、schema、DAL、SDK API、UIKit / UI 行为、测试或生产代码 | `./tools/run_all_tests.sh` | 全量脚本已包含协议生成、文档一致性校验、后端测试、E2E、前端 unit / sdk / ui |
| 数据库 schema 设计调整 | 先征求确认，再跑 `./tools/run_all_tests.sh` | 研发阶段直接修改当前 schema，不写 migration、`ALTER TABLE` 或旧数据兼容 |

> 全量脚本会自动按需安装 Playwright 浏览器并对网络环节重试，正常无需手动操作。
> 若需在新机器上补齐系统依赖（`libnss` 等），可在 `frontend/` 目录执行：
>
> ```bash
> npx playwright install --with-deps chromium
> ```
>
> 或运行全量脚本时设置 `YIMSG_PLAYWRIGHT_WITH_DEPS=1`。

### 3.1 测试提速（不降低强度）

全量脚本在保持测试覆盖与断言完全不变的前提下做了以下提速，核心思路是「去掉重复构建 + 降低与被测逻辑无关的固定成本 + 提高互不干扰用例的并行度」：

- **bcrypt 哈希成本**：脚本统一导出 `YIMSG_BCRYPT_COST=4`（`bcrypt.MinCost`），由 `internal/auth` 在进程启动时读取并覆盖默认的 `DefaultCost`。哈希算法与校验逻辑完全不变，仅去掉生产级抗暴力成本，使大量注册场景（Go 单测 `internal/service`、E2E、SDK 集成、Playwright 种子）的单次哈希从 ~67ms 降到 ~1ms。生产部署不设置该变量，保持 `DefaultCost`。
- **构建产物复用**：脚本只构建一次前端 `web/` 和服务端二进制；Playwright `globalSetup` 通过 `YIMSG_SKIP_FRONTEND_BUILD=1` 跳过重复的 `npm run build`，通过 `YIMSG_PREBUILT_SERVER` 复用脚本已编译的服务端二进制，避免重复 `go build`。单独运行 `npm run test:ui` 时不设置这两个变量，会自动回退到本地构建。
- **UI 并行度**：`playwright.config.ts` 默认 `workers: '100%'`（按 CPU 核数并行，Playwright 默认仅用一半核数）。UI 用例各自创建独立用户、互不共享状态，可用 `PLAYWRIGHT_WORKERS` 覆盖（数字或百分比）。

上述变量仅作用于测试链路，不改动任何 schema、接口或生产默认配置。

### 3.2 网络健壮性（沙箱 / CI 优先）

在受限代理与沙箱网络下，下载类步骤最易因瞬时中断导致整轮失败。全量脚本针对此做了如下处理，目标是在不降低测试强度的前提下提升成功率与速度：

- **网络步骤统一重试**：`npm ci`、Playwright 浏览器安装、`go install protoc-gen-go`、`go mod download` 均走指数退避重试（2s/4s/8s/16s），可用 `YIMSG_NET_RETRY` 覆盖最大尝试次数（默认 4）。
- **Playwright 下载单次超时 + 自动切镜像**：每次 `playwright install chromium` 尝试都用 `timeout` 包裹，单次超过 `YIMSG_PLAYWRIGHT_DOWNLOAD_TIMEOUT`（默认 150s）未完成就直接放弃、转入下一次重试，不再傻等官方源自然超时/断连。调用方未显式设置 `PLAYWRIGHT_DOWNLOAD_HOST` 时，第 2 次起自动切换到 `npmmirror.com` 镜像；显式设置了该变量则全程尊重用户选择、不覆盖。
- **依赖已具备则跳过**：`protoc-gen-go` 命中目标版本即跳过安装；Playwright 通过实际无头启动探针判断浏览器是否就绪，就绪则不再联网安装，显著缩短重复运行耗时。
- **改用完整 chromium 构建**：`playwright.config.ts` 的 chromium 项设置 `channel: 'chromium'`，以完整 chromium 构建运行新版无头模式，不依赖单独的 `chrome-headless-shell` 二进制——后者是额外大体积下载，在受限代理下最易被中断。因此 Playwright 安装**以浏览器能否启动为成败判据**，而非 `playwright install` 的退出码，避免被 `chrome-headless-shell` 下载失败误伤整轮 UI 测试。
- **可选跳过浏览器安装**：设置 `PLAYWRIGHT_SKIP_BROWSER_DOWNLOAD=1` 或 `YIMSG_SKIP_PLAYWRIGHT_INSTALL=1` 时跳过浏览器安装（适用于已预装匹配版本的环境）。
- **Windows 下浏览器安装目录固定到 `%LOCALAPPDATA%` 之外**：部分 Windows 环境的 `%LOCALAPPDATA%` 会被宿主打包机制静默重定向（例如指向 `...\Packages\<PackageFamilyName>\LocalCache\Local\...`），普通文件 API 能透明读到重定向后的内容，但 `chrome.exe` 依赖的 side-by-side 清单加载走另一套路径解析，找不到该目录下的清单文件，导致启动报 `"side-by-side configuration is incorrect"`（与系统是否装有 VC++ 运行库无关）。全量脚本在 Windows 下、调用方未显式设置 `PLAYWRIGHT_BROWSERS_PATH` 时，固定改用 `C:/ms-playwright-browsers` 安装浏览器，避开该重定向。

## 5.1 CI 质量趋势指标（覆盖率 + 重复率）

作为 CI 质量门禁和趋势观察口径，提供一组可重复执行的指标：

| 维度 | 工具 | 口径 | 配置入口 | 当前基线 | 阈值 |
|---|---|---|---|---|---|
| Go 单测行覆盖率 | `go test -coverprofile` + `go tool cover -func` | `internal/...` 生产代码包总体 | `tools/check_go_coverage.sh` | **58.6%** | 55% |
| 前端 SDK 单测覆盖率 | `vitest --coverage`（`@vitest/coverage-v8`） | `frontend/src/sdk/**` | `frontend/vitest.config.ts` 的 `coverage.thresholds` | lines 66.9% / branches 50.5% / functions 65.4% / statements 64.9% | lines 55 / branches 50 / functions 55 / statements 55 |
| 代码重复率 | `jscpd` | `internal/`、`cmd/`、`frontend/src/` 中的 `.go` / `.ts`，排除测试与生成产物 | `.jscpd.json`（`threshold=3`） | 1.21% 行 / 0.97% token | 3% |

执行入口：

```bash
# 一次性执行 Go 覆盖率 + 前端覆盖率 + 重复率，任意一项不达标则退出非零：
./tools/check_quality.sh

# 也可以并入全量脚本（默认不执行以避免拖慢本地迭代）：
YIMSG_QUALITY_GATES=1 ./tools/run_all_tests.sh
```

执行口径补充：

- Go 覆盖率仅统计 `internal/...`；`cmd/server/main.go` 作为进程入口由 E2E 黑盒覆盖，不计入阈值。
- 前端覆盖率仅统计 UI 无关的 `src/sdk/**`；`uikit/` / `views/` / `worker/` 主要由 Playwright UI 测试与 SDK 集成测试覆盖，不计入单测阈值。
- 重复率启用 `mode: strict`、`minTokens: 60`、`minLines: 10`，避免把 Go 导入块、短常量列表误判为克隆；阈值参考 jscpd 推荐的"至少一个函数体"规模，后续需要收紧可独立下调 `minTokens`。
- 首次引入时阈值保守起步；**阈值只允许逐步上调**，若特殊情况下需临时下调，必须在本文件同一 PR 留痕说明。
